## 1. 总体结论

建议采用三层结构：

```text
应用监督层（SRT）
  bm_module：命令、状态机、故障、通信、诊断、参数持久化
        │ event / snapshot
        ▼
领域控制层（HRT + SRT）
  bm_exec：电机轴、功率级、BMS Pack、温控区等独立实例
        │ state + config + resources
        ▼
纯算法层（无框架状态、无 HAL）
  PID/PI、滤波、FFT/频谱、斜坡、坐标变换、SVPWM、积分器、估算器
```

职责边界：

| 层 | 应放内容 | 不应放内容 |
|----|----------|------------|
| 纯算法层 | 确定性数学计算、显式状态、显式配置 | HAL、事件、日志、全局可变状态 |
| 领域控制层 | 采样标定、算法编排、PWM/ADC/编码器访问、安全停机 | 通信协议、Shell、持久化 |
| 应用监督层 | 模式切换、故障锁存、命令处理、诊断 | 高频闭环、逐采样数据流 |

**关键规则**：算法核是普通 C API；可独立运行的控制对象才是
`bm_exec`；需要 SRT 生命周期的业务单元才是 `bm_module`。

### 1.1 技术深度与成熟度要求

本路线图的目标不是堆叠教学算法，也不能停留在 SimpleFOC/sample FOC 一类
“理想采样 + 基础变换 + PI + SVPWM 能转起来”的入门层。示例可以浅，但公共
领域组件必须持续推进到真实硬件非理想、边界工况、故障、标定和量产诊断。

算法的**实现顺序**与**成熟度**是两个正交维度。Phase A–D 表示先做什么，
下表表示做到什么程度：

| 标识 | 阶段 | 定义 | 可使用范围 |
|------|------|------|------------|
| `D0` | 机制演示 | 验证调度、HAL、snapshot 或 API 接线；算法可为桩或理想化模型 | 学习、CI 冒烟，禁止产品声明 |
| `E1` | 前期应用探索 | 算法闭环成立，有参考模型和基本边界测试，但目标硬件、工况和参数覆盖有限 | 原型机、实验台、可行性研究 |
| `V2` | 工程验证 | 明确目标硬件与运行包络，完成非理想补偿、故障注入、PIL/HIL、WCET 和资源预算 | 工程样机、小批验证 |
| `I3` | 工业成熟 | 多硬件/多工况长期验证，具备标定、降额、诊断、生产测试、版本兼容和现场数据闭环 | 在声明包络内量产使用 |

`I3` 表示“在已声明的硬件、参数和工况包络内有工业应用证据”，不表示自动
获得 IEC 61508、ISO 26262、IEC 62304 等功能安全认证。

成熟度标识必须绑定到**具体组件版本 + 数值后端 + 目标平台 + 工况包络**。
不能笼统声称“FOC 已工业成熟”；应写成类似：

```text
motor_foc_sensored 1.2 / float32 / STM32G4
24-60 V, 5-20 kHz PWM, 双电阻采样, 编码器反馈：V2
```

当前仓库的 `bm_algorithm`（K0 纯数学核）已作为公共库交付，整体成熟度为
`E1`（前期应用探索），详见 §0 与 [api/bm_algorithm.md](api/bm_algorithm.md)。

`hrt_servo_stub`、`hrt_bms_coulomb`、`multi_channel_bms` 和
`multi_axis_sync` 仍属于 `D0`：它们证明框架执行与通信机制，不证明对应控制
算法达到 `E1/V2/I3`，且尚未接线公共算法库。

### 1.2 成熟度晋级门槛

| 晋级 | 必须增加的证据 |
|------|----------------|
| `D0 → E1` | 方程/离散化说明、参考模型、闭环指标、基本异常输入、至少一套真实或可信 plant |
| `E1 → V2` | 目标包络、硬件非理想补偿、最坏工况、故障注入、PIL/HIL、WCET/Flash/RAM、参数标定流程 |
| `V2 → I3` | 多批硬件与环境覆盖、长稳/寿命数据、生产测试、诊断与降额、参数追溯、升级回滚、现场问题闭环 |

任何阶段均不得以“Demo 可以运行”“电机可以转”“曲线看起来正常”替代晋级
证据。成熟度下降也必须被允许：目标 MCU、传感器拓扑、PWM 频率或编译器发生
实质变化时，原有等级需重新评估。

### 1.3 物理问题优先原则

公共算法不按“公式是否实现”验收，而按“是否解决声明的物理问题”验收。
每个算法必须绑定对象、传感链、执行器、非理想、故障、标定、指标和运行包络。
完整契约与已有算法族的补强要求见
[24 §2–3](24-物理世界领域与算法深度矩阵.md#2-物理问题追溯契约)。

这条规则适用于本文已经列出的全部算法，包括 PI/PID/PR、滤波、FFT、FOC、
SVPWM、观测器、EKF/UKF、SOC/SOH、MPPT/PLL、轨迹、AGC/VAD、视觉算子和
TinyML 前后处理，不能只约束以后新增的算法。

## 3. 建议代码布局

第一阶段只引入一个独立、可裁剪的公共算法库。算法范围以
[24 §3.2](24-物理世界领域与算法深度矩阵.md#32-权威算法清单) 为权威清单；
本节只定义物理布局：

```text
include/
  bm_algorithm.h                 # 聚合头，按配置暴露算法
  bm/algorithm/
    bm_algo_common.h             # 饱和、死区、滞环、速率限制
    bm_algo_control.h            # integrator、PI/PID、PR、补偿器
    bm_algo_filter.h             # LPF/HPF、FIR、biquad、notch
    bm_algo_profile.h            # ramp、梯形/S 曲线
    bm_algo_motion.h             # 编码器、DDA、运动辅助
    bm_algo_motor.h              # 变换、SVPWM 等纯数学核
    bm_algo_power.h              # PLL、MPPT、计量纯数学核
    bm_algo_battery.h            # SOC/SOH 模型与估算核
    bm_algo_fft.h                # CFFT/RFFT、窗、幅度/功率谱
    bm_algo_spectral.h           # PSD、Goertzel、相关、包络
    bm_algo_resample.h           # 抽取、插值和重采样
    bm_algo_fusion.h             # 姿态、固定维度估算核
    bm_algo_audio.h              # PCM/EQ/AGC/VAD 数学核
    bm_algo_image.h              # 低分辨率基础算子
    bm_algo_features.h           # TinyML 特征与量化辅助
Source/
  algorithm/                     # 与公共头按算法族对应
tests/unit/
  algorithm/                     # 参考向量、边界和多实例测试
```

领域组件先放在示例或应用侧验证，成熟后再提升为可选库：

```text
components/
  control_loop/
  motion_profile/
  motor/
  power/
  bms/
  sensor_quality/
  estimation_fusion/
  spectral_diagnostics/
  stream_frontend/
  process_sequence/
  transport_qos/
  fault_derating/
extensions/
  external_algorithm/            # 外部库/协处理器静态适配

```

不建议一开始就在 `Source/` 中建立四套完整领域库。先稳定纯算法 API，
再从真实 Demo 中提炼领域组件，可以减少错误抽象。未适配领域统一遵循
[24 §5.2](24-物理世界领域与算法深度矩阵.md#52-未适配领域的静态扩展接口)。

### 3.1 领域组件晋级规则

领域组件从 `Demo/` 或产品应用提升到公共 `components/` 前，至少满足：

- 已被两个独立实例或两个应用复用，不包含板级单例；
- HAL 依赖全部通过 `instance->resources` 注入；
- 配置、状态、命令和遥测结构有明确所有者；
- 有 native_sim 参考模型、故障测试和目标板 WCET 数据；
- 公共 API 不暴露具体 MCU 寄存器、句柄布局或协议字段。

提升到公共 `components/` 只代表达到最低 `E1`，不代表工业成熟。每个组件
目录应提供 `MATURITY.md`，至少记录：

- 当前等级、版本、数值后端、目标 MCU/板卡和编译器；
- 已验证/未验证的电压、电流、速度、温度、频率等运行包络；
- 已知限制、禁用工况、残余风险和降级行为；
- MIL/SIL/PIL/HIL 报告、WCET/资源报告和硬件测试索引；
- 从上一级晋级仍缺少的证据。

## 4. 纯算法 API 约束

### 4.1 状态与配置显式分离

```c
typedef struct {
    float kp;
    float ki;
    float out_min;
    float out_max;
    float integrator_min;
    float integrator_max;
} bm_algo_pi_config_t;

typedef struct {
    float integrator;
    float output;
} bm_algo_pi_state_t;

void bm_algo_pi_reset(bm_algo_pi_state_t *state, float output);
float bm_algo_pi_step(bm_algo_pi_state_t *state,
                      const bm_algo_pi_config_t *config,
                      float error,
                      float dt_s);
```

统一约束：

- 无内部全局可变状态，天然支持多实例；
- 不分配内存，不调用 HAL、事件、日志或临界区；
- 循环次数有静态上界，HRT 路径不使用递归；
- 输入、输出、积分状态均有饱和和异常策略；
- `reset`、`step`、必要时 `validate_config` 命名一致；
- 算法函数不通过函数指针做数值后端分派，避免增加 HRT 抖动。

### 4.2 数值类型策略

建议分两步：

1. 首批提供 `float` 实现，面向 Cortex-M4F/M7、ESP32 和 PC 仿真。
2. API 稳定后增加显式的 Q31/Q15 版本，例如
   `bm_algo_pi_q31_step()`，不使用全局 typedef 在编译期偷偷切换 ABI。

原因：

- 浮点版更容易建立参考向量和验证算法正确性；
- 显式后缀让 Flash、精度和溢出行为可审计；
- 8 位 Ultra/Nano 不应被迫链接 FOC、biquad 等 Control 级算法。

### 4.3 时间参数

- 通用算法接受显式 `dt_s` 或离散系数，不读取框架时钟；
- 固定周期 HRT 可在 `init` 时预计算离散系数；
- `step` 中避免除法、三角函数和系数重算；
- FOC 的 sin/cos 可先使用查表或平台加速接口，平台优化不得改变公共算法状态布局。

### 4.4 单位、方向与缩放约定

算法 API 发布前必须统一以下约定，否则不同组件即使类型兼容也可能计算错误：

| 项目 | 必须明确 |
|------|----------|
| 物理量 | SI、工程单位或 per-unit；禁止同一 API 混用 |
| 角度 | 弧度/机械圈/电角度、零点、正方向、归一化区间 |
| Clarke/Park | 幅值不变或功率不变缩放，abc 相序与 dq 符号 |
| PWM | duty 的满量程、中心/边沿对齐、最小脉宽和死区补偿 |
| 时间 | 秒、微秒或预离散系数；结构字段名带单位后缀 |
| 定点 | Q 格式、舍入、饱和、累加器位宽和移位顺序 |

建议公开类型和字段使用 `current_a`、`period_us`、`angle_rad` 等单位后缀；
HAL 原始计数只在领域适配层出现，不直接进入算法核。

### 4.5 异常与返回值

- 配置合法性在 `init` / `validate_config` 阶段完成，HRT `step` 不重复做重校验；
- `step` 的异常策略须区分“输出饱和”和“输入无效”，不能都静默夹紧；
- 需要报告状态的算法返回 `status + output`，或在 state 中保存状态字；
- 浮点版明确 NaN/Inf、负零、非规格化数和 `-ffast-math` 策略；
- 定点版禁止依赖有符号整数溢出，所有中间结果使用显式位宽和饱和函数；
- 使用 `sqrtf/sinf/cosf` 等 `libm` 能力时，须记录链接成本和目标工具链差异。

### 4.6 FFT 与频谱算法约束

FFT 属于通用 DSP 基础能力，之前路线图按闭环控制链展开而遗漏，应补入
`bm_algorithm`。首批建议范围：

- radix-2 complex FFT 与 real FFT，先支持 64/128/256/512/1024 点；
- Hann/Hamming/Blackman 窗及 coherent gain、ENBW 修正；
- magnitude、power、PSD、单边谱归一化；
- Goertzel 单频/少频检测，避免少量频点也运行完整 FFT；
- 谱峰插值、RMS、峰值、crest factor、包络等诊断特征；
- 后续增加 overlap STFT、实数卷积和定点 Q15/Q31 后端。

API 必须显式传入工作区，禁止内部堆分配：

```c
typedef struct {
    uint32_t size;
    float *work;
    uint32_t work_count;
    const float *twiddle;
} bm_algo_rfft_f32_t;

int bm_algo_rfft_f32_init(bm_algo_rfft_f32_t *fft,
                          uint32_t size,
                          float *work,
                          uint32_t work_count);
int bm_algo_rfft_f32_execute(bm_algo_rfft_f32_t *fft,
                             const float *time_data,
                             float *spectrum);
```

实现与测试必须明确：

- FFT 正/逆方向、缩放、bin 顺序、Nyquist/DC 表示和原地/非原地规则；
- 输入长度必须为支持的静态上界，初始化阶段拒绝非法长度；
- twiddle/bit-reversal 表可编译期生成或只读共享，不在 HRT 中生成；
- 窗函数造成的幅值和噪声带宽修正不能留给调用者猜测；
- Q15/Q31 每级缩放、block floating 或饱和策略必须成为 API 契约；
- FFT 只给出频谱估计，频率分辨率和泄漏由采样率、点数、窗和重叠共同决定。

## 5. 与现有框架的组合方式

### 5.1 一个伺服轴

```text
mod_servo_supervisor (bm_module, SRT)
  ├─ 订阅 enable/disable/setpoint/fault 事件
  ├─ 发布命令 snapshot
  └─ 读取 telemetry snapshot

servo_axis_0 (bm_exec)
  ├─ Hardware slot: ADC → Clarke/Park → PI(d/q) → SVPWM → PWM
  ├─ Scheduled slot: encoder → speed PI → iq_ref
  ├─ state: FOC/PI/observer 的全部可变状态
  ├─ config: 电机参数、增益、限幅、采样周期
  ├─ resources: PWM/ADC/ENC/COMP 句柄
  └─ claims: PWM、ADC、ENC、COMP、IRQ
```

多轴时复制 `state/config/resources/claims`，复用同一套 `ops` 和
`slots`。算法函数不感知轴编号。

### 5.2 一个数字电源通道

```text
power_stage_0 (bm_exec)
  Hardware HRT: 电流 PI / 峰值限流 / PWM 更新
  Scheduled HRT: 电压 PI / 软启动 / 前馈

mod_power_supervisor (bm_module)
  SRT: OFF → PRECHARGE → SOFT_START → RUN → FAULT
  负责参数命令、故障锁存与恢复策略
```

多相电源的各相为独立实例，通过 `bm_sync` 配置相位；均流控制放在一个
较慢的 Scheduled HRT 协调实例中。

### 5.3 一个 BMS Pack

```text
pack_sampler (bm_exec)
  Hardware HRT: ADC/DMA 完成、标定、快照发布

pack_estimator (Scheduled HRT 或 SRT module)
  库仑积分、SOC 融合、温度补偿

mod_bms_supervisor (bm_module)
  过压/欠压/过温去抖、故障锁存、均衡策略、通信
```

SOC 估算频率通常不需要 Hardware HRT。EKF 等计算量较大的算法应先测
WCET，再决定放慢速 Scheduled HRT 还是分片到 SRT。

### 5.4 HAL 前置能力

当前 HAL 足以演示“读 ADC、写 duty、绑定 IRQ”，但完整控制算法还需要先
补齐或由板级资源层明确承担以下契约：

| 能力 | 典型用途 |
|------|----------|
| PWM 周期/满量程、中心对齐、互补输出、死区、影子寄存器提交 | FOC、PFC、LLC |
| ADC 触发相位、规则/注入组、DMA 帧、采样时间和校准 | 同步采样、BMS 多通道 |
| 采样时间戳与 PWM 周期序号 | 多速率新鲜度、延迟补偿 |
| 编码器计数模数、零位/index、捕获时间 | 速度估算、角度展开 |
| 比较器/Break 原因、锁存查询和受控恢复 | 故障诊断与安全重启 |
| 平台数学加速能力声明 | CORDIC、DSP 指令、饱和算术 |
| I2S/SAI/PDM/DAC、Codec、DMA half/full block 回调 | 音频输入输出与连续块处理 |
| DMA buffer 对齐、cache clean/invalidate、一致性屏障 | Cortex-M7/多核/带 cache 平台 |

不宜把这些配置硬编码进算法。`resources` 保存 HAL 句柄，板级初始化负责
触发拓扑，领域组件在 `init` 中校验分辨率、量程和采样周期是否满足算法要求。

FOC 还必须明确单/双/三电阻电流重构策略；数字电源必须明确 PWM 到 ADC 的
采样延迟；BMS 必须明确 DMA 帧完整性和通道映射。这些均属于领域适配层。

### 5.5 实例内并发与状态所有权

同一 `bm_exec` 的 Hardware HRT 和 Scheduled HRT 可能处于不同中断
优先级，快环可以抢占慢环。两者共享一个 `instance->state` 不等于可以无保护
地读写同一字段。

建议把状态按所有者拆开：

```text
current_state     仅 Hardware HRT 写
speed_state       仅 Scheduled HRT 写
command_box       SRT 写、HRT 读
fast_to_slow_box  快环写、慢环读
telemetry_box     HRT 写、SRT 读
```

- 快慢环交换复合数据使用 snapshot 或明确的单写者 mailbox；
- 对齐标量也必须说明原子性、`volatile` 和内存屏障需求；
- 禁止快慢环同时更新同一个 PI、滤波器或故障状态结构；
- Hardware HRT 抢占 Scheduled HRT 的时间计入慢环 WCET；
- 多个 Hardware IRQ 可能访问同一实例时，须由 HAL 优先级设计或领域层拒绝该配置。

### 5.6 生命周期编排

`bm_module` 与 `bm_exec` 是并列子系统，当前框架不会因为启动模块而自动
启动控制实例。应用必须选择一个且仅一个控制批次所有者：

**推荐方式 A：由 `main` 编排**

```text
module boot
→ ctrl init
→ 可选 sync configure/arm
→ ctrl start
→ 可选 sync trigger
→ main loop
→ ctrl safe_stop
→ module stop/deinit
```

**方式 B：应用侧适配模块**

定义唯一 `mod_control_runtime`，由它在 `start/stop` 中管理整张实例表，适合
希望所有应用功能都出现在模块表中的项目。该适配模块只负责批次生命周期，
不能为每个轴各自调用 `bm_exec_init_all()`。

无论哪种方式，都应保证：

- 事件类型和监督模块先就绪，再开放控制输出；
- 任一控制启动失败都保持 PWM 安全态；
- stop 顺序先关闭命令入口，再 `bm_exec_safe_stop_all()`；
- `bm_exec_init_all()` 会替换既有会话，禁止多个模块争用控制批次所有权；
- 控制实例表、同步域和监督模块的依赖顺序写入应用文档和测试。

### 5.7 音频与块式 DSP 集成

框架可覆盖以下资源受限音频场景：

- 提示音、告警音、语音对讲前处理和轻量音效；
- 麦克风/振动传感器频谱分析、设备异音与轴承故障检测；
- DC blocker、EQ、FIR/IIR、mix、gain、limiter、AGC、轻量 VAD；
- 小规模麦克风阵列 delay-and-sum、声源能量/方向特征；
- FFT/STFT 特征提取后交给上层规则或轻量模型。

不应默认覆盖：

- MP3/AAC/Opus 等完整媒体编解码栈；
- USB Audio、Bluetooth A2DP/LE Audio 等协议；
- 大通道数、高采样率制作、复杂混响和大型神经降噪/语音识别。

推荐执行路径：

```text
I2S/SAI/PDM/DAC + DMA ping-pong
  → half/full IRQ 只确认块、更新时间戳/序号
  → 有界 audio block callback 处理静态 buffer
  → 输出块提交 DMA
  → 频谱特征/告警通过 snapshot 或小事件送 SRT
```

48 kHz 音频不能逐采样发布 `bm_event`，也不能把 PCM 流塞入 `bm_snapshot`。
`bm_snapshot` 只适合传递最新特征和状态，不适合保存连续样本历史。

当前 `BM_EXEC_SLOT_HARDWARE` 已通过 `bind` 解耦具体事件源，但回调仍只接收
实例指针；`bm_channel` 又明确禁止在 HRT ISR 使用。因此正式音频支持前应
新增独立 `bm_audio_stream` /
`bm_dsp_stream` 组件及 HAL 契约，而不是把 I2S DMA 生硬伪装成 ADC slot：

- 静态 ping-pong/ring buffer 与 buffer ownership 状态机；
- DMA half/full、underrun/overrun、clock loss 和 sample slip 计数；
- block size、sample rate、channel layout、sample format 显式配置；
- ISR 只做常数时间交接；DSP 链运行上下文和优先级由平台明确；
- 每块处理 WCET 必须小于块周期，并保留更高优先级控制 ISR 的抢占预算；
- 音频与电机/电源控制共核运行时，控制安全路径优先，音频允许丢块或降级。

音频块可以建模为一个 `bm_exec` 的资源与生命周期，但流调度应由专用
stream 组件承担；增益、EQ、FFT 等算法核仍保持无 HAL 的普通 C API。

## 6. 命令、遥测与故障通道

| 数据方向 | 内容 | 机制 |
|----------|------|------|
| SRT → HRT | 设定值、模式、限幅、参数版本 | 反向 `bm_snapshot` |
| HRT → SRT | 最新测量、控制输出、状态字 | `bm_snapshot` |
| HRT → SRT | 过流、采样失败、deadline miss | `publish_copy_from_isr` 发小事件 |
| SRT → SRT | 通信命令、诊断、状态迁移 | `bm_event` / `bm_channel` |
| 亚微秒故障 | 短路、硬过压 | COMP/PWM Break 硬件链路 |

运行时参数更新建议使用“双配置 + 版本号”：

1. SRT 在非活动配置中完成全部字段写入和范围校验；
2. 通过 snapshot 发布新配置及递增版本；
3. HRT 在周期边界一次性切换 active config；
4. 结构体不得包含指向 SRT 临时缓冲区的指针。

### 6.1 数据帧与新鲜度

跨域 payload 不应只有裸数值。测量、命令和遥测建议携带：

```c
typedef struct {
    uint32_t sequence;
    uint32_t timestamp_ticks;
    uint32_t status;
    float value;
} app_sample_t;
```

- HRT 检查命令版本是否变化、命令是否超时；
- 慢环检查快环数据的 sequence/timestamp，拒绝重复或过期样本；
- 时间戳必须来自单调时基，并定义回绕比较规则；
- telemetry 明确有效位、饱和位、降额位和故障位。

`bm_snapshot` 是单写者、单读者三缓冲。多个 SRT 消费者不能直接共享同一
snapshot 的 `reading` 状态；应由一个监督模块读取后通过事件/通道扇出，
或为每个读者提供独立 snapshot。payload 拷贝时间也必须计入 HRT WCET。

### 6.2 故障模型

领域组件应统一故障分类，而不是只提供一个 `fault` 布尔值：

| 类别 | 示例 | 处理 |
|------|------|------|
| 硬件立即停机 | 短路、硬过压、驱动器故障 | Break/COMP 先关断，软件只记录 |
| 控制周期停机 | ADC 无效、位置丢失、数值异常 | 当前或下一 HRT 周期请求安全态 |
| 降额运行 | 过温预警、母线偏低 | 限流/限功率并上报 |
| SRT 锁存 | 通信超时、重复启动失败 | 状态机进入 FAULT，显式确认后恢复 |

每个故障定义检测域、响应上限、锁存方式、恢复条件和遥测编码。恢复前必须
验证硬件锁存已清除、命令已撤销且算法状态已 reset，禁止直接沿用旧积分量。

### 6.3 标定与参数持久化

- ADC offset/gain、相序、编码器零位等标定值属于产品配置，不属于算法常量；
- 持久化参数包含 schema 版本、长度、CRC 和单位版本；
- 上电先校验范围与交叉约束，再生成 active config；
- 参数更新失败时保留上一份已知良好配置；
- 安全关键限值与普通调参项分区，运行中修改权限不同；
- PI/PID 增益变更须定义积分状态保持、缩放或 reset 的无扰切换策略。

## 7. CMake 与配置规划

新增独立目标，默认关闭领域算法，基础算法可按 profile 启用：

```cmake
option(BM_ENABLE_ALGORITHM "Enable deterministic control algorithms" OFF)

add_library(bm_algorithm STATIC ...)
target_link_libraries(bm_algorithm PUBLIC bm_config)
```

建议的 profile：

| Profile | 算法 |
|---------|------|
| `minimal` / `event` | 关闭 |
| `control` | common、control、filter、profile、calibration |
| `servo` | `control` + motion、motor |
| `power` | `control` + power、battery 可选 |
| `dsp` | filter、FFT、spectral、resample、statistics |
| `fusion` | filter、calibration、fusion |
| `media_edge` | `dsp` + audio/image/features；依赖 stream/pipeline |
| `full` | 仅用于开发验证，不作为产品默认 profile |

细粒度裁剪使用配置宏，例如：

```c
#define BM_CONFIG_ENABLE_ALGO_PI          1
#define BM_CONFIG_ENABLE_ALGO_FILTER      1
#define BM_CONFIG_ENABLE_ALGO_TRANSFORM   0
#define BM_CONFIG_ENABLE_ALGO_SVPWM       0
#define BM_CONFIG_ENABLE_ALGO_FFT         0
#define BM_CONFIG_ENABLE_ALGO_FUSION      0
#define BM_CONFIG_ENABLE_ALGO_AUDIO       0
#define BM_CONFIG_ENABLE_ALGO_IMAGE       0
```

公共入口建议保持分层：

- `#include "bm_algorithm.h"`：只引入算法；
- `#include "zeplod.h"`：框架运行时，不默认强制引入全部算法；
- 应用可同时包含两者。

这样 Nano 项目不会因为统一入口而扩大编译和 API 表面积。

构建还应明确：

- `bm_algorithm` 不依赖 `bm_core`，只依赖 `bm_config` 和 C 标准库；
- 使用 `libm` 的源文件独立成目标或按功能链接，避免基础 PI 被迫拉入三角函数；
- float 与 Q31/Q15 使用不同源文件和符号，可在同一固件中并存；
- 编译选项记录 FPU ABI、浮点收缩、fast-math 和溢出相关选项；
- `tools/list_sources.py`、静态库安装清单和 size 测量同步支持算法目标。

## 8. 实现优先级

### Phase A：通用确定性算法核

优先实现：

1. saturate/clamp、deadband、hysteresis、rate limiter；
2. integrator、PI、PID（抗积分饱和、无扰切换）；
3. 一阶低通、滑动平均、中值/异常值抑制、biquad/notch；
4. ramp、梯形速度轨迹；
5. 导数/速度估算、单位和角度归一化辅助函数。

这批算法可同时服务电源、电机、BMS、温控，复用价值最高。

### Phase B：运动与电机基础

1. Clarke/Park 与逆变换；
2. SVPWM 与占空比限幅；
3. 电流采样 offset 校准与单/双/三电阻重构；
4. 编码器速度估算和角度展开；
5. 步进梯形/S 曲线脉冲规划与微步电流参考；
6. 电流/速度双环参考组件；
7. `Demo/foc_current_loop`，先做有位置传感器 FOC。

首版不要直接做无感 FOC。无感启动、观测器切换和低速稳定性会显著扩大
验证范围，应在有感 FOC 和 WCET 基线稳定后进入 Phase D。

### Phase C：数字电源与 BMS

1. Buck/Boost 双环、软启动、前馈和保护限幅；
2. PFC 电流整形、LLC 频率控制与多相均流；
3. P&O 与增量电导 MPPT；
4. 库仑计量的时间尺度修正、偏置补偿；
5. OCV-SOC 查表与温度插值；
6. 过压/欠压/过温去抖、接触器/预充和被动均衡策略。

### Phase D：高级估算与控制

1. SOGI/PLL、PR 控制器；
2. BMS EKF；
3. 无感 FOC 观测器、MTPA、弱磁；
4. S 曲线、多轴插补、阻抗控制。

高级算法须按目标 MCU 分别建立精度、RAM、Flash 和 WCET 门槛。

### Phase E：流式 DSP、音频与视觉

1. `bm_block` / `bm_stream` 静态零拷贝块流；
2. FFT/RFFT、窗、PSD、Goertzel、相关、重采样；
3. 音频 gain/EQ/mix/limiter/AGC/VAD；
4. 振动包络、阶次跟踪和诊断特征；
5. IMU 融合、超声 ToF、功率质量；
6. 低分辨率图像转换、阈值、形态学、连通域；
7. TinyML 特征与静态 arena 适配。

Phase E 依赖 [23](23-多领域确定性流式架构.md) 的 block/frame 执行模型，
不能仅靠 `bm_event`、`bm_snapshot` 或现有 `bm_channel` 实现连续数据流。

### Phase F：外部生态与未适配领域

1. 静态 extension descriptor 与 ABI/version 校验；
2. process-image 和 bounded transport 适配；
3. 外部 algorithm node 与 accelerator provider；
4. 外部处理器心跳、失联、bypass 和 safe-stop；
5. safety monitor 边界接口，只承担非安全通道协作；
6. 每类接口至少两个 provider 后再考虑进入公共 ABI。

Phase F 不以“支持某协议或外部库”为目标，而以稳定的数据所有权、时钟、
资源、deadline 和故障边界为验收对象。

### 8.1 电机控制深度基线

| 阶段 | 技术范围 |
|------|----------|
| `E1` | 坐标变换、基础 d/q PI、SVPWM、有感角度反馈、基本电流 offset 校准、闭环 plant 指标 |
| `V2` | 单/双/三电阻重构、采样窗口有效性、母线前馈、交叉耦合补偿、抗饱和、死区/器件压降补偿、编码器异常、启动/停机/反转、过调制边界、弱磁或限压、参数失配测试 |
| `I3` | 温度与母线全包络、MTPA/弱磁平滑切换、转矩纹波与噪声优化、在线/离线参数辨识、降额、传感器失效策略、生产相序/零位/电流校准、多电机批次差异、长稳与现场诊断 |

无感 FOC 还必须单独覆盖：

- 静止定位、开环爬升、观测器接管和失败回退；
- 低速可观测性、过零/反转、再捕获和顺风启动；
- 参数温漂、负载突变、母线跌落与弱磁区稳定性；
- BEMF/SMO/PLL/磁链观测器的相位延迟和离散噪声；
- 观测角度可信度、失锁检测和安全停机。

只实现 Clarke/Park、PI 和 SVPWM 的组件应命名为 `foc_core` 或
`foc_experimental`，成熟度最高只能标记为 `E1`，不得称为完整工业 FOC。

### 8.2 数字电源深度基线

| 阶段 | 技术范围 |
|------|----------|
| `E1` | 指定拓扑的小信号/平均模型、基本电流/电压环、软启动、限幅、离线负载阶跃 |
| `V2` | CCM/DCM 或工作区切换、采样与计算延迟、PWM/DAC 量化、输入/母线前馈、抗积分饱和、斜坡补偿、轻载 burst/skip、启动/关断时序、短路/过压/欠压/磁饱和保护、环路稳定性测量 |
| `I3` | 输入/负载/温度/器件公差全包络、拓扑模式无扰切换、效率/热/EMI 协同、多相均流与掉相、生产校准、老化漂移、故障降额、长稳和电网/负载异常数据闭环 |

特定拓扑还需增加：

- PFC：过零畸变、输入电压前馈、PLL/SOGI、谐波/THD、brown-in/out；
- LLC：增益曲线、谐振参数偏差、频率限制、容性区规避、同步整流时序；
- MPPT：辐照突变、局部阴影、搜索振荡、功率限额和 DC-link 协调；
- 多相 VRM：相位管理、动态加减相、电流共享、相故障隔离。

### 8.3 BMS 深度基线

| 阶段 | 技术范围 |

| `E1` | 带时间戳的库仑积分、静态 OCV-SOC 表、基本温度补偿、阈值/滞环/去抖、被动均衡原型 |
| `V2` | 电流 offset 漂移与零电流校准、休眠电流、容量/效率修正、OCV 静置判定与迟滞、SOC 融合/EKF 可观测性、采样开线/粘连/不一致诊断、预充/接触器/粘连检测、均衡热约束 |


`closed_loop_servo` 用 native_sim 构造一阶电机模型：

```text
速度设定 → ramp → PI → duty → 一阶 plant → 速度反馈
```

它同时验证：

- 算法多实例和 reset 行为；
- Scheduled HRT 周期执行；
- SRT 命令 snapshot 与 HRT 遥测 snapshot；
- `bm_module` 监督层启动、停机和故障事件；
- 不依赖真实电机即可形成可重复回归测试。

该包的目标成熟度是 `E1`，不是工业伺服。完成后，再用相同 PI/filter/ramp
API 进入 FOC 和数字电源；各领域组件按 §8.1–§8.4 独立晋级。

## 10. 测试与验收

### 10.1 每个算法必须覆盖

- 零输入、阶跃、正负饱和、恢复、reset；
- NaN/Inf 策略（浮点版）和整数溢出策略（定点版）；
- 配置边界及非法参数；
- 固定参考向量，与 Python/Octave 离线结果比较；
- 两个状态实例交错调用，证明无隐藏共享状态。

浮点断言使用绝对误差和相对误差组合，不直接比较位相等。

### 10.2 领域组件必须覆盖

- init/start/safe_stop 失败路径；
- 资源 Claim 冲突；
- Hardware/Scheduled 槽的调用频率；
- 命令版本切换和 snapshot 一致性；
- ADC/HAL 错误立即进入安全输出；
- deadline miss、过流和通信超时后的状态迁移。

### 10.3 分层验证

| 层级 | 目标 |
|------|------|
| 单元/参考向量 | 单个算法数值正确性与边界 |
| MIL | Python/Octave plant 与控制器闭环指标 |
| SIL/native_sim | 实际 C 算法、模块、snapshot 和故障注入 |
| QEMU | 架构、ISR 接线和构建回归；不宣称硬件 WCET |
| PIL/HIL | 目标编译器数值差异、真实时序、ADC/PWM/保护链路 |

除示例波形外，还应检查可量化指标：超调、稳态误差、调节时间、相位裕度、
THD/纹波、SOC 误差等。float 与定点实现使用同一组黄金向量做差异比较。

### 10.4 发布门槛

| 维度 | 门槛 |
|------|------|
| 正确性 | 单元测试 + 参考向量全部通过 |
| 确定性 | 无堆、无递归、循环上界明确 |
| 实时性 | 目标 MCU 给出平均/最大周期和 WCET 余量 |
| 资源 | 记录各算法 Flash/RAM 增量 |
| 安全 | 饱和、错误输入、HAL 失败均有定义行为 |
| 文档 | API、执行域、单位、采样周期和数值范围明确 |
| 可追溯 | 需求 ID、测试 ID、版本与参考模型可对应 |

### 10.5 成熟度标注规范

每个算法或领域组件必须在以下位置显示相同的成熟度：

1. 公共头文件 Doxygen：`@maturity D0/E1/V2/I3`；
2. 组件 `README.md` 和 `MATURITY.md`；
3. 示例列表和 API 文档；
4. 发布说明与测试报告。

推荐组件文档头：

```text
Maturity: E1 - 前期应用探索
Validated: float32 / native_sim + STM32G4 / 20 kHz / 双电阻
Not validated: 无感、弱磁、过调制、-20~85 C 全温区
```

标注规则：

- 未提供标识时按 `D0` 处理；
- `experimental`、`exploration` 对应 `E1`，不得使用“production ready”；
- `V2` 必须链接目标硬件测试、PIL/HIL、WCET 和已知限制；
- `I3` 必须声明有效包络和证据版本，禁止脱离平台单独宣传；
- API 稳定性与算法成熟度分开记录，稳定 API 也可能只有 `E1`；
- 任一已知问题突破安全或稳定边界时，应立即降低等级并记录原因。

CI 可检查公共 `components/` 是否存在 `MATURITY.md`，但成熟度判定本身必须
经过领域评审，不能由测试数量或覆盖率百分比自动生成。

## 11. 不建议的集成方式

- 不要把 `pi_step()`、`lpf_step()` 各自注册为 `bm_module`；
- 不要让算法核直接读取全局 ADC 或写全局 PWM；
- 不要在 HRT 中发布高频完整采样数据到事件队列；
- 不要用一个巨型 `motor_control.c` 同时承担算法、HAL、通信和故障状态机；
- 不要先设计通用矩阵/插件系统，再寻找实际控制场景；
- 不要用运行时函数指针切换 float/Q31 后端。

这些做法会削弱多实例能力、WCET 可分析性和资源裁剪效果。

## 13. 实施前检查清单

开始某个领域组件前，应能回答：

1. 输入输出单位、坐标系、方向、量程和更新频率是什么？
2. 谁拥有 state/config/resources，是否满足单写者规则？
3. 哪些 HRT slot 会互相抢占，是否存在同字段并发访问？
4. 快慢环之间如何判断数据新鲜、超时和 sequence 回绕？
5. 谁是控制批次生命周期的唯一所有者？
6. HAL 是否提供所需触发、时间戳、原子 PWM 提交与硬件保护？
7. 最坏错误输入、HAL 失败和 deadline miss 时输出是什么？
8. 参数如何校验、持久化、升级、回滚和无扰切换？
9. float/定点数值误差、Flash/RAM、平均周期和 WCET 预算是多少？
10. 参考模型、黄金向量、SIL/PIL/HIL 各验证什么？

以上任一项没有明确答案时，组件仍处于实验性 Demo 阶段，不应提升为公共
领域库。
