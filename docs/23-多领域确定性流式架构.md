# 23 多领域确定性流式架构

> **本文职责**：把 Zeplod Baremetal 从控制专用执行底座扩展为资源受限 MCU
> 的确定性控制、流式 DSP 与边缘采集框架，并给出直接重构方案。
> **当前状态**：架构路线图。现有已实现能力仍以 [01](01-框架概览与资源层级.md)
> 和公共头文件为准。
> 行业场景、真实物理痛点和算法深度要求见
> [24](24-物理世界领域与算法深度矩阵.md)。

## 1. 定位

建议将长期定位调整为：

> 面向资源受限 MCU 的可裁剪确定性事件、控制与流式计算框架。

框架不试图成为 RTOS、Linux 媒体栈或通用计算图运行时。它提供的是：

- 静态内存与有界执行；
- 中断、周期、DMA 块/帧和协作式后台四类执行路径；
- 显式资源、缓冲区、时钟域和数据所有权；
- 控制、DSP、采集、诊断算法的多实例生命周期；
- 从机制演示到工业成熟的证据化成熟度管理。

现有机电控制能力是第一类核心应用，而不是未来唯一应用。

## 2. 可覆盖领域

### 2.1 原生适配

| 领域 | 典型节点 | 核心执行模式 |
|------|----------|--------------|
| 电机/运动/机器人 | 伺服、ESC、步进、关节 | IRQ 单步 + 周期环 + SRT |
| 数字电源/能源 | DC-DC、PFC、LLC、逆变、MPPT | PWM/ADC IRQ + 多速率环 |
| BMS/储能 | Pack 采集、SOC/SOH、均衡、预充 | DMA/ADC 块 + 周期估算 + SRT |
| 音频 DSP | 对讲前端、提示音、EQ、AGC、VAD | DMA 音频块 + 有界 DSP 链 |
| 状态监测 | 振动、异音、轴承、结构健康 | 采样块 + FFT/包络/特征 |
| 工业测量 | 示波采集、记录仪、功率质量、计量 | 同步采样块 + 统计/频谱 |
| 传感融合 | IMU/AHRS、姿态、导航前端 | 时间戳样本 + 周期融合 |
| 超声/声呐/低速雷达 | 测距、液位、存在检测 | 激励同步 + ADC 块 + 相关/FFT |

### 2.2 条件适配

| 领域 | 可覆盖范围 | 主要缺口 |
|------|------------|----------|
| 低分辨率机器视觉 | 灰度、阈值、形态学、连通域、线/码检测、简单光流 | DCMI/CSI、帧缓冲、外部 RAM、硬件 JPEG |
| 视频端点 | 低帧率传感、帧统计、OSD、硬件编解码外围调度 | 软件 H.264/H.265 和高分辨率管线不适合中小 MCU |
| 生物信号 | ECG/PPG/EMG 前处理、心率/呼吸率、伪迹检测 | 医疗认证、临床算法证据属于产品侧 |
| 通信 DSP | DTMF/FSK、Goertzel、匹配滤波、同步、轻量 FEC | 完整蜂窝/Wi-Fi/复杂 SDR 协议栈 |
| TinyML 前后处理 | 特征、量化、静态 arena 调度、异常检测 | 推理内核建议接 CMSIS-NN/TFLM 等外部库 |
| 智能照明/显示 | LED 电流、调光、色彩变换、小型段码/点阵 | 复杂 GUI 和高分辨率显示栈 |
| 热/过程控制 | 温控、泵阀、炉控、流量/压力 | 工业协议和产品安全逻辑 |
| 触觉与人机输入 | 力反馈、触觉波形、按键/触摸滤波 | GUI、USB/HID 等协议按需外置 |

### 2.3 明确不覆盖

- 桌面/移动端通用音视频媒体框架；
- 高分辨率软件视频编解码与复杂 GUI；
- 动态插件、动态计算图和依赖堆的模型运行时；
- 大规模 SDR、基带协议栈和操作系统网络栈；
- 未经产品认证即可宣称医疗、航空或 ASIL-D 安全能力。

### 2.4 系统级覆盖边界

框架“覆盖某领域”只表示能够承担其中的确定性 MCU 子系统，不表示提供完整
行业产品：

- PLC、工业网关、EtherCAT/PROFINET 需要外部语言 runtime 或认证协议栈；
- AGV、农机和机器人可覆盖底盘/关节实时层，SLAM、规划和安全控制器外置；
- 医疗、轨交、汽车高安全场景只能复用非安全机制，不能继承认证结论；
- 超声成像、高分辨率视频、5G 基带和金属增材高速协调通常需要 FPGA/SoC。

逐场景判定见 [24 §4](24-物理世界领域与算法深度矩阵.md#4-行业场景覆盖矩阵)。

## 3. 当前结构性限制

| 当前机制 | 优点 | 多领域限制 |
|----------|------|------------|
| `bm_event` | 小消息、SRT 发布订阅 | 不适合 PCM、图像、ADC 连续帧 |
| `bm_snapshot` | 最新值、单写单读 | 不保存历史，不能表达帧队列和多消费者 |
| `bm_channel` | SRT SPSC 拷贝队列 | HRT 禁用，大块拷贝成本高 |
| `bm_exec` | 通用执行实例、资源校验、Hardware/Periodic 槽 | 尚未表达 block/frame 输入和 deadline |
| `bm_hrt` | 周期 ISR 调度 | 不表达 DMA block deadline |
| 现有 HAL | PWM/ADC/COMP/ENC 完整度较高 | 缺 I2S/SAI/PDM/DAC/DCMI、通用 DMA stream |

`bm_exec` 已移除 PWM/ADC 触发枚举依赖；具体事件源由 `bind` 适配器连接。
后续扩展应增加 block/frame 执行契约，而不是重新引入外设类型枚举。

## 4. 目标执行模型

```text
IRQ-Step RT       每次硬件事件处理一个确定性控制 step
Periodic RT       按固定周期执行慢环、融合、监督计算
Block/Frame RT    DMA half/full 或 frame-ready 驱动有界块处理
Cooperative SRT   事件、协议、状态机、诊断、存储
```

| 执行类 | 数据粒度 | 典型期限 | 示例 |
|--------|----------|----------|------|
| IRQ-Step | 标量/小向量 | 微秒级 | FOC、电源电流环 |
| Periodic | 状态/小批量 | 100 us–秒 | 速度环、AHRS、SOC |
| Block RT | 16–2048 样本 | 下一个 DMA 块到达前 | 音频、FFT、超声 |
| Frame RT | 行/帧 | 下一行/帧到达前 | 低分辨率视觉 |
| SRT | 消息/命令 | 无硬截止 | 协议、日志、参数 |

Block/Frame RT 不是普通 SRT。它允许比 IRQ-Step 更长的处理，但仍有确定的
块周期 deadline、静态工作区和过载策略。

## 5. 直接核心重构

### 5.1 `bm_exec` 通用执行实例

`bm_exec` 已直接取代旧控制实例组件，不保留旧头文件、类型别名或旧构建选项。
当前公共描述符为：

```c
typedef enum {
    BM_EXEC_SLOT_HARDWARE,
    BM_EXEC_SLOT_PERIODIC
} bm_exec_slot_kind_t;

typedef struct bm_exec bm_exec_t;

typedef struct {
    bm_exec_slot_kind_t kind;
    uint32_t period_us;
    void (*run)(const bm_exec_t *inst);
    int (*bind)(const bm_exec_t *inst,
                const bm_hal_hrt_binding_t *binding);
    const char *name;
} bm_exec_slot_t;
```

关键变化：

- 当前核心只理解 Hardware/Periodic，不理解 ADC、I2S 或 Camera；
- `bind` 由 HAL/领域适配器连接具体事件源；
- 生命周期、状态、配置、资源 Claim 由 `bm_exec_t` 统一承载；
- block/frame 输入将在 `bm_stream` 契约稳定后直接扩展 `bm_exec`；
- 旧 API 不进入弃用期，仓库内调用点已一次性迁移。

### 5.2 `bm_stream` 静态零拷贝块流

```c
typedef struct {
    void *data;
    uint32_t capacity_bytes;
    uint32_t valid_bytes;
    uint32_t sequence;
    uint32_t timestamp_ticks;
    uint16_t format;
    uint16_t flags;
} bm_block_t;
```

`bm_stream` 管理静态 block descriptor 和预分配 payload：

```text
FREE → DMA_OWNED → READY → PROCESSING → FREE
                  └→ OUTPUT_READY → DMA_OWNED
```

首版契约：

- 单生产者、单消费者；
- 零堆、零拷贝交接；
- ISR 只提交 descriptor，不复制 payload；
- 支持 acquire/commit/peek/release；
- 支持 overrun、underrun、drop、late、corrupt 计数；
- 策略可选 drop-newest、drop-oldest、mute/bypass、safe-stop；
- payload 对齐、cache 一致性和 DMA ownership 由 HAL 契约明确。

`bm_stream` 与 `bm_channel` 不合并：前者服务 HRT DMA 大块所有权，后者服务
SRT 小对象拷贝。

### 5.3 静态 DSP/媒体管线

新增可选 `bm_pipeline`，使用编译期静态 DAG 或线性链：

```c
typedef struct {
    int (*prepare)(void *state, const void *config);
    int (*process)(void *state,
                   const bm_block_t *input,
                   bm_block_t *output);
    void (*reset)(void *state);
    const char *name;
} bm_pipeline_node_ops_t;
```

约束：

- 节点和边在编译期定义；
- 不允许运行时增删节点；
- 每个节点声明输入输出格式、工作区、最坏计算量；
- 初始化时校验格式、buffer 数和 deadline 预算；
- 运行时支持 bypass、reset 和故障隔离；
- 首版只支持线性链和单输入单输出，再扩展有限 fan-out/fan-in。

控制领域继续直接使用 step，不必为了统一而强制转换成 block pipeline。

### 5.4 通用资源标识

现有 `bm_resource_kind_t` 是闭合枚举，长期应改为可扩展 ID：

```c
typedef uint16_t bm_resource_class_t;

#define BM_RESOURCE_CLASS_CORE_BASE   0x0000u
#define BM_RESOURCE_CLASS_HAL_BASE    0x0100u
#define BM_RESOURCE_CLASS_APP_BASE    0x8000u
```

内置 class 可包含 TIMER、DMA、IRQ、MEMORY_REGION、CLOCK_DOMAIN、
STREAM_ENDPOINT、ACCELERATOR。旧 PWM/ADC 等枚举映射为兼容 class。

### 5.5 时钟域与时间戳

多媒体和融合算法必须知道数据来自哪个时钟：

```c
typedef struct {
    uint16_t clock_id;
    uint16_t quality;
    uint32_t ticks;
    uint32_t rate_hz;
} bm_timestamp_t;
```

需要支持：

- 单调回绕比较；
- 采样时钟与系统时钟映射；
- audio sample slip / drift 计数；
- 多 ADC、IMU、Camera 的时间对齐；
- `bm_sync` 从 PWM 相位同步扩展为通用 clock/trigger group，但不承担重采样。

## 6. 数据机制选型

| 数据 | 机制 |
|------|------|
| 小命令、故障通知 | `bm_event` |
| 最新状态/特征 | `bm_snapshot` |
| SRT 小对象序列 | `bm_channel` |
| DMA PCM/ADC/图像块 | `bm_stream` |
| 算法节点连接 | `bm_pipeline` + `bm_stream` |
| 参数与校准 | 版本化 config snapshot + 持久化模块 |

禁止用同一个万能消息总线承载所有数据。

## 7. HAL 扩展

新增通用 HAL 契约：

| HAL | 能力 |
|-----|------|
| `bm_hal_dma_stream` | buffer 提交、half/full、方向、对齐、错误 |
| `bm_hal_audio` | I2S/SAI/PDM/DAC、sample rate、format、channels、clock |
| `bm_hal_camera` | DCMI/CSI、line/frame、pixel format、crop |
| `bm_hal_capture` | 通用 ADC/SPI 并行采集块、trigger、timestamp |
| `bm_hal_accel` | DSP/CORDIC/CRC/crypto/2D 等能力查询和提交 |
| `bm_hal_cache` | DMA clean/invalidate、cache line 和一致性 |

协议栈与 Codec 寄存器配置可放在 board/应用模块，不强制进入核心 HAL。

### 7.1 HAL 之外的领域扩展边界

HAL 只抽象 MCU 外设能力，不承载 PLC 对象模型、协议报文、神经网络 runtime
或安全认证语义。未适配领域使用 [24 §5.2](24-物理世界领域与算法深度矩阵.md#52-未适配领域的静态扩展接口)
定义的静态 extension：

- `STREAM_ENDPOINT` 连接 DMA、外部 ADC、Camera 或协处理器数据面；
- `ALGORITHM_NODE` 连接外部 AEC、Codec、TinyML 和视觉算法；
- `ACCELERATOR` 连接 DSP/CORDIC/NPU/FPGA；
- `PROCESS_IMAGE` 连接 PLC 和实时以太网协议栈；
- `TRANSPORT` 连接 CAN/Ethernet/Wi-Fi/LoRa 栈；
- `SAFETY_MONITOR` 只与独立安全岛交换 permit/trip/heartbeat。

extension descriptor 在初始化期注册和校验，不允许运行时加载动态插件。具体
协议栈、厂商句柄和模型对象只存在于 provider 私有结构中。

### 7.2 扩展接口与执行模型的组合

| 外部能力 | 数据面 | 执行位置 | 故障时动作 |
|----------|--------|----------|------------|
| 外部算法库 | `bm_block` 或显式小向量 | pipeline node / Periodic RT | bypass、冻结输出或 safe-stop |
| FPGA/NPU/协处理器 | stream endpoint + accelerator job | Block/Frame RT | cancel、丢块、降级本地链 |
| 工业协议栈 | process image + bounded transport | SRT + 同步提交点 | 输出失效、过程镜像安全态 |
| 上位 SLAM/规划 | timestamped command snapshot | SRT 接收，HRT 消费 | 新鲜度超时、减速或停车 |
| 独立安全 MCU | safety monitor | 独立硬件路径 | trip 优先，普通软件不得覆盖 |

所有跨处理器命令都必须携带 sequence、时间戳、有效期和来源状态；HRT 不直接
阻塞等待外部处理器、总线或加速器完成。

## 8. 算法族

### 8.1 通用 DSP

- FIR、IIR/biquad、notch、重采样、DC blocker；
- CFFT/RFFT、窗、PSD、STFT、Goertzel；
- 卷积/相关、Hilbert/包络、RMS、峰值、crest factor；
- 统计、直方图、异常值、标定与插值；
- float32、Q31、Q15 显式后端。

### 8.2 音频

- gain/mix/pan、EQ、compressor/limiter、noise gate；
- AGC、VAD、AEC 前置接口、PDM decimation；
- delay-and-sum、GCC-PHAT 小阵列定位；
- tone/DTMF、提示音合成；
- TinyML 特征：log-mel、MFCC。

复杂 AEC、神经降噪和 Codec 可接外部库，框架负责静态 buffer 与 deadline。

### 8.3 图像/视频前处理

- RGB/YUV/gray 转换、crop/resize、gamma；
- threshold、morphology、Sobel、积分图；
- 连通域、质心、线/边缘、简单 barcode/marker；
- 帧差、背景模型、有限光流；
- TinyML 图像量化和 tensor 前后处理。

### 8.4 测量、通信与感知

- 锁相、同步检波、matched filter、AGC；
- FSK/DTMF/脉冲检测、CRC/FEC 基础算法；
- 超声 ToF、互相关、包络、峰值；
- 雷达 range FFT、静态 clutter 去除、小规模 Doppler；
- 功率计量 RMS/P/Q/THD、谐波和事件检测；
- IMU calibration、complementary/Mahony/Madgwick/EKF 接口。

## 11. 重构阶段

### R0：纠正定位

- README 区分当前实现与多领域路线图；
- 所有 Demo 保持 `D0`；
- FFT/DSP/音视频算法进入路线图。

| 条目 | 状态 |
|------|------|

- 固化 extension ABI/version、能力和 workspace 查询；
- 为 stream、algorithm、accelerator、process image、transport 各提供
  native_sim fake provider；
- 选择两个真实 provider 验证公共接口不是单厂商抽象；
- safety monitor 只验证边界和故障优先级，不宣称获得安全等级。

## 12. 首批代码落地建议
