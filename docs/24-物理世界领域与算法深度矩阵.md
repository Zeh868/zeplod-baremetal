# 24 物理世界领域与算法深度矩阵

> **本文职责**：逐项审计 Zeplod Baremetal 可覆盖的行业场景，并把算法名称
> 追溯到真实对象、物理非理想、失效模式、验收指标和成熟度证据。
> **不负责**：重复执行域、流式缓冲和 API 接线，相关设计见 [22](22-领域算法与模块化路线图.md)
> 与 [23](23-多领域确定性流式架构.md)。

## 1. 结论与边界

附件中的场景不能统一理解为“框架能够完整提供该产品”。应分为四级：

| 级别 | 含义 | Zeplod 承担范围 |
|------|------|-----------------|
| `A` 原生闭环 | MCU 内确定性采样、估算、控制和保护是主要矛盾 | 可承担主要实时内核和领域组件 |
| `B` 流式前端 | 主要矛盾是 DMA 块/帧、固定延迟和有界 DSP | 待 `bm_stream` / Block RT 后承担前端 |
| `C` 集成底座 | 核心价值在协议、调度、认证或系统级协同 | 只承担本地控制、采集、整形和适配 |
| `D` 非目标整机 | 吞吐、安全等级或生态超出中小 MCU 框架边界 | 仅可作为子节点，禁止整机能力宣称 |

`A/B` 也不等于工业成熟。所有新领域默认从 `D0` 或 `E1` 开始，只有具体组件
在声明包络内完成 [22 §1.2](22-领域算法与模块化路线图.md#12-成熟度晋级门槛)
要求，才可晋级 `V2/I3`。

## 2. 物理问题追溯契约

文档中出现的每个领域算法，在进入公共 `bm_algorithm` 或 `components/` 前，
必须回答以下问题：

1. **对象是什么**：电机、功率级、电芯、液压缸、声场、光学成像链或通信信道。
2. **传感器测到什么**：单位、量程、带宽、噪声、偏置、安装误差、时间戳来源。
3. **执行器能做什么**：饱和、死区、速率限制、迟滞、量化、最小脉宽和安全态。
4. **主要非理想是什么**：温漂、老化、机械间隙、时钟漂移、采样延迟、EMI、污染。
5. **算法解决哪个痛点**：降低超调、抑制谐振、补偿漂移、减少漏检或保证同步。
6. **失败时会发生什么**：错误输出、失稳、热失控、机械碰撞、误报警或数据丢失。
7. **如何验收**：误差、稳定时间、频响、THD、SNR、误报漏报、WCET、丢块率。
8. **在哪个包络有效**：目标 MCU、编译器、采样率、电压/温度/负载和传感拓扑。

只有数学公式和理想输入测试的算法最高为 `E1`。没有物理非理想、故障注入、
标定和目标硬件证据，不得标记为 `V2/I3`。

## 3. 已有算法的深度补强

以下要求覆盖 [22](22-领域算法与模块化路线图.md) 和
[23](23-多领域确定性流式架构.md) 已描述的算法，不允许只实现教科书主路径。

| 算法族 | 要解决的真实痛点 | 必须覆盖的非理想与工程指标 |
|--------|------------------|----------------------------|
| PI/PID/PR | 跟踪误差、负载扰动、周期波动 | 输出/积分饱和、anti-windup、无扰切换、D 项噪声、变 `dt`、执行器速率限制、闭环裕度 |
| ramp/梯形/S 曲线 | 冲击、失步、结构振动、液压冲击 | 速度/加速度/jerk/转矩联合约束、离散终点误差、急停距离、在线重规划 |
| LPF/FIR/IIR/biquad/notch | 传感噪声、开关纹波、机械共振 | 相位延迟、系数量化、极点稳定性、饱和、启动瞬态、频率漂移和动态 notch 更新 |
| FFT/RFFT/PSD/STFT | 谐波、轴承故障、异音、功率质量 | 抗混叠、ADC ENOB、窗增益/ENBW、泄漏、重叠、时钟抖动、频率分辨率、块 deadline |
| Goertzel/同步检波 | 少量频点、载波或导频检测 | 频偏、幅相标定、邻频干扰、检测门限、积分时间和误报漏报 |
| 相关/matched filter/ToF | 弱回波、超声测距、同步捕获 | 激励振铃、声速温补、多径、旁瓣、门限、亚采样插值、不可检测状态 |
| Clarke/Park/SVPWM | 三相电流解耦和母线利用率 | 相序/极性、offset/gain、采样窗口、死区、器件压降、母线纹波、过调制和最小脉宽 |
| FOC 观测器/PLL | 无传感器角度、转速和再捕获 | 低速不可观、参数温漂、相位延迟、失锁、顺风启动、反转、弱磁、可信度与回退 |
| EKF/UKF/互补融合 | 偏置漂移、多传感器异步和短时失真 | 可观测性、协方差调参、时间对齐、异常值门控、传感器隔离、延迟补偿和坐标安装误差 |
| OCV-SOC/SOH/库仑积分 | 电流积分漂移、容量衰减和静置修正 | offset、休眠电流、库仑效率、OCV 迟滞、温度/倍率/老化、容量与内阻在线更新 |
| MPPT/SOGI/PLL | 辐照变化、并网同步和谐波限制 | 局部阴影、搜索振荡、限功率、过零畸变、频偏、弱网、THD 和 DC-link 协调 |
| AGC/compressor/limiter/VAD | 音量跨度、削波、噪声触发和听感 | attack/release、噪声泵动、定点噪声、爆音、端到端延迟、语料误检漏检 |
| AEC/ANC/阵列前端 | 回声、主动降噪和声源方向 | 声学路径变化、次级通道、因果性、扬声器非线性、双讲、时钟漂移；高级实现可外接 |
| 图像阈值/形态学/连通域 | 反光、污渍、照度变化和目标粘连 | 自动曝光、非均匀照明、镜头畸变、运动模糊、ROI、误检漏检和帧 deadline |
| TinyML 特征/量化 | 规则难覆盖的异常和关键词 | 数据漂移、类别不平衡、量化误差、arena 峰值、置信度、OOD、版本回滚和数据闭环 |
| 去抖/滞环/限值监控 | 开关抖动、噪声越限和报警风暴 | 时间/计数去抖、迟滞、传感可信度、冻结值、变化率、告警锁存和恢复条件 |
| 标定/插值/统计 | 批次差异、零点和增益漂移 | 标定点覆盖、单调性、外推策略、温度分段、CRC/版本、生产追溯和不确定度预算 |
| 优先级队列/整形 | 网关突发、关键报文被大流量阻塞 | 有界队列、丢弃策略、token bucket、deadline、背压、时间戳和拥塞统计 |

### 3.1 可集成算法分级

算法清单按其对框架基础设施的依赖分为四类，后续路线图、CMake profile 和
组件目录均以本节为准：

| 标识 | 含义 | 进入仓库的条件 |
|------|------|----------------|
| `K0` 纯算法核 | 无 HAL、无框架状态、显式 state/config/workspace | 可直接进入 `bm_algorithm`，优先实现 |
| `K1` 确定性组合 | 由多个 `K0` 算法组成，依赖 `bm_exec`、snapshot 或现有 HAL | 进入 `components/`，按领域独立成熟 |
| `K2` 流式/帧式算法 | 依赖 `bm_block`、`bm_stream`、时间戳或 pipeline | 等 [23](23-多领域确定性流式架构.md) R1/R2 后集成 |
| `K3` 外部能力适配 | 算法主体来自协议栈、加速器、外部库或更高算力处理器 | 只提供静态适配接口，不复制完整外部生态 |

`K0-K3` 表示集成形态，不表示成熟度。每项仍需独立标注 `D0/E1/V2/I3`。

**当前可用性**（2026-06-13）：

| 组件 | 状态 | 成熟度 | 说明 |
|------|------|--------|------|
| `bm_algorithm` / `Source/algorithm/` | **已交付** | `E1` | K0 纯数学核，float32；见 [22 §0](22-领域算法与模块化路线图.md#0-仓库实现进度)、[api/bm_algorithm.md](api/bm_algorithm.md) |
| `bm_stream` / `bm_block` | **部分交付** | `E1` 探索 | 静态零拷贝块流与单元测试已有；与 `bm_exec` block slot、DMA HAL 的深度集成待 M3/M4 |
| `components/`、`extensions/` | 未开始 | — | 领域组件与外部适配仍按 Demo 侧验证 |

[24 §3.2](#32-权威算法清单) 中标记为 `K0` 且已列入 `bm_algorithm` 的条目可视为**公共 API 可用**；`K1`–`K3` 及未实现的清单项仍为规划状态。发布说明中标注 “available” 须同时满足：公共头、源文件、基本单元测试与成熟度记录。

### 3.3 暂缓或仅保留接口的算法

以下算法不进入首批公共实现：动态维度矩阵与通用优化器、全身 MPC、完整
SLAM/图优化、高分辨率波束形成与成像重建、软件视频编解码、完整 5G 基带、
复杂神经 AEC/降噪、认证安全表决器。原因不是算法价值不足，而是其内存、
吞吐、生态或认证责任超出当前框架边界。它们统一通过 §5.2 的扩展接口接入。

## 4. 行业场景覆盖矩阵

### 4.1 工业控制、制造与物流

| 场景 | 级别 | 真实痛点与算法链 | 框架边界 |
|------|------|------------------|----------|
| PLC / 软 PLC / 分布式 I/O | `C` | 输入抖动和异步到达 → 时间戳输入镜像、去抖、IEC 定时器/计数器、顺序联锁、PID、输出镜像原子提交、扫描超时监控 | 可做确定性执行核；IEC 61131-3 语言、在线调试、冗余和工业网络运行时外置 |
| 伺服、CNC、多轴运动 | `A` | 电流采样窗口、机械共振、轮廓误差 → FOC、速度/位置环、前馈、notch、S 曲线、电子齿轮、交叉耦合轮廓控制 | 当前最匹配；工业以太网主站和高阶 CNC 规划器外置 |

## 5. 可复用领域组件包

不应为 37 个场景复制 37 套算法。优先建立以下可组合组件：

| 组件包 | 解决的问题 | 复用领域 |
|--------|------------|----------|
| `control_loop` | PI/PID/PR、anti-windup、前馈、限幅、无扰切换 | 电机、电源、液压、HVAC、灌溉 |
| `motion_profile` | 梯形/S 曲线、DDA、同步、急停和在线重规划 | CNC、机器人、AGV、电梯、3D 打印 |
| `sensor_quality` | 标定、去抖、冻结/变化率、异常值、冗余一致性 | BMS、农机、医疗前端、消防、过程控制 |
| `estimation_fusion` | 时间对齐、bias、互补/EKF、传感隔离 | 飞控、AGV、农机、可穿戴 |
| `spectral_diagnostics` | FFT/PSD/包络/阶次/谱峰和质量指标 | 振动、异音、电网、DAQ |
| `stream_frontend` | DMA block、重采样、时钟漂移、overrun/underrun | 音频、医疗信号、超声、采集 |
| `process_sequence` | 版本化状态机、联锁、超时、ramp/soak | PLC、充电、半导体、消防 |
| `transport_qos` | 有界队列、整形、deadline、新鲜度和拥塞统计 | 工业网关、CAN 桥接、实时以太网适配 |
| `fault_derating` | 故障锁存、降额、恢复条件和证据快照 | 电机、电源、BMS、液压、充电 |

### 5.1 组件目录与依赖方向

```text
bm_algorithm              K0：纯数学核，只依赖 bm_config/C 标准库
    ↑
components/*              K1：领域编排，可依赖 bm_exec/snapshot/HAL
    ↑
bm_stream + bm_pipeline   K2：块/帧所有权、deadline、格式协商
    ↑
extensions/*              K3：外部库、协议栈、协处理器和安全旁路适配
```

依赖只能向下，不允许 `bm_algorithm` 回调框架，也不允许外部协议字段进入通用
算法状态。K3 适配层必须能被完全关闭，不改变 K0/K1 的 ABI。

### 5.2 未适配领域的静态扩展接口

扩展机制不是动态插件系统。所有 descriptor、workspace 和 endpoint 均由应用
静态定义，在 `init` 阶段完成 ABI、能力、格式、资源和 deadline 校验。

#### 统一扩展描述符

```c
typedef uint16_t bm_extension_class_t;

typedef struct {
    uint16_t abi_major;
    uint16_t abi_minor;
    bm_extension_class_t extension_class;
    uint32_t capability_bits;
    const char *name;
    int (*validate)(const void *config, const void *resources);
    int (*prepare)(void *state, const void *config, const void *resources);
    void (*safe_stop)(void *state);
} bm_extension_descriptor_t;
```

约束：

- `abi_major` 不匹配时拒绝启动，`abi_minor` 只允许向后兼容增加能力；
- `validate/prepare` 仅在初始化或受控重配置阶段调用；
- 高频 `step/process` 不放入统一描述符，由具体扩展类定义，避免万能回调；
- 所有内存、资源 Claim、时钟域和线程/中断上下文必须由调用者显式提供；
- `safe_stop` 必须幂等，外部处理器失联时有本地降级或旁路策略。
- K3 默认不得进入 IRQ-Step RT；确需进入时必须改为编译期直接绑定，并单独
  提供可重入性、最坏周期和故障终止时间证据。

#### 六类扩展契约

| 扩展类 | 最小接口 | 服务的未适配领域 |
|--------|----------|------------------|
| `BM_EXT_STREAM_ENDPOINT` | `acquire/commit/release`、format、timestamp、ownership | 高速 DAQ、Camera、超声、外部 ADC |
| `BM_EXT_ALGORITHM_NODE` | `query_workspace/prepare/process/reset`、固定最大 shape | AEC/ANC、编解码、TinyML、视觉库 |
| `BM_EXT_ACCELERATOR` | capability query、submit/poll/cancel、cache fence | DSP/CORDIC/NPU/2D/FPGA |
| `BM_EXT_PROCESS_IMAGE` | 输入/输出镜像、sequence、freshness、atomic commit | PLC、EtherCAT/PROFINET 从站 |
| `BM_EXT_TRANSPORT` | bounded RX/TX、priority、deadline、link state | CAN/Ethernet/Wi-Fi/LoRa 网关 |
| `BM_EXT_SAFETY_MONITOR` | heartbeat、permit、trip、reason、evidence snapshot | ASIL/SIL/医疗项目中的独立安全岛 |

外部算法节点建议使用固定 shape 契约：

```c
typedef struct {
    uint16_t input_format;
    uint16_t output_format;
    uint32_t max_input_bytes;
    uint32_t max_output_bytes;
    uint32_t workspace_bytes;
    uint32_t worst_case_cycles;
    uint32_t flags;
} bm_ext_algorithm_caps_t;

typedef struct {
    int (*query_caps)(const void *config, bm_ext_algorithm_caps_t *caps);
    int (*prepare)(void *state, void *workspace, uint32_t workspace_bytes);
    int (*process)(void *state,
                   const bm_block_t *input,
                   bm_block_t *output);
    void (*reset)(void *state);
} bm_ext_algorithm_ops_t;
```

如果尚未实现 `bm_block`，K3 适配器只能停留在设计或应用私有层，不能临时用
`bm_event` 传输大块数据。

#### 未适配领域映射

| 领域缺口 | 接口组合 | 本框架保留的责任 |
|----------|----------|------------------|
| 完整 PLC runtime/工业以太网 | `PROCESS_IMAGE + TRANSPORT` | 本地 I/O、控制、镜像一致性、超时安全态 |
| SLAM/全身规划/大型优化 | `STREAM_ENDPOINT + ALGORITHM_NODE` | 采集时间戳、低层控制、命令新鲜度和急停 |
| 高级 AEC/神经音频/Codec | `ALGORITHM_NODE + ACCELERATOR` | DMA 流、格式、buffer、deadline 和 bypass |
| 高分辨率视觉/超声重建 | `STREAM_ENDPOINT + ACCELERATOR` | 触发、行/帧交接、元数据和过载策略 |
| CMSIS-NN/TFLM/NPU | `ALGORITHM_NODE + ACCELERATOR` | 特征、arena 上限、模型版本、置信度和降级 |
| FPGA/SoC/高速 DSP 协处理 | `STREAM_ENDPOINT + ACCELERATOR + TRANSPORT` | 控制面、时钟映射、心跳和失联恢复 |
| ASIL-D/SIL4/植入式医疗 | `SAFETY_MONITOR`，且独立硬件/软件链 | 仅非安全功能；不得替代独立安全决策 |

### 5.3 扩展接口晋级门槛

扩展接口进入公共头文件前必须具备两个不同 provider 的验证样例；只有一个
厂商实现时先放 `extensions/vendor_or_project/`。公共扩展还必须提供：

- native_sim fake provider 和异常注入；
- ABI/能力/格式不匹配测试；
- workspace 不足、deadline miss、外部失联和 reset 测试；
- cache/DMA ownership、调用上下文和重入性说明；
- 一个完全关闭该扩展的最小构建，证明无反向依赖和 Flash 泄漏。

## 6. 领域组件的验收模板

每个组件的 `MATURITY.md` 除 [22 §3.1](22-领域算法与模块化路线图.md#31-领域组件晋级规则)
要求外，还应包含：

```text
Physical plant:
Sensor chain:
Actuator chain:
Operating envelope:
Timing/deadline:
Non-idealities covered:
Faults and degraded modes:
Calibration/EOL procedure:
Acceptance metrics:
Reference model/dataset:
Target MCU/compiler/backend:
WCET/Flash/RAM:
Known unsafe or unsupported cases:
Maturity: D0 | E1 | V2 | I3
```

验收指标必须与物理结果相关。例如：

- 运动控制报告跟随误差、轮廓误差、超调、停止距离和同步偏差；
- 电源报告环路裕度、负载阶跃、THD、效率、温升和保护动作时间；
- BMS 报告 SOC/SOH 误差分布、收敛时间、异常电芯检出和误报警；
- 音频报告延迟、频响、THD+N、SNR、丢块和主观异常事件；
- 诊断/视觉/TinyML 报告数据包络、误报、漏报、置信度和漂移；
- 网关报告端到端延迟分位数、deadline miss、队列水位和各优先级丢包。

## 7. 扩展领域成熟度基线

### 7.1 PLC、过程与顺序控制

| 阶段 | 技术范围 |
|------|----------|
| `E1` | 单设备 plant、固定扫描/周期、去抖/联锁/PID/超时成立，能检测扫描超期 |
| `V2` | 纯滞后、迟滞、传感冻结、执行器卡滞、模式切换、掉电恢复、参数版本和故障注入 |
| `I3` | 多设备/季节/负载长期数据、现场整定、冗余传感策略、生产配置追溯和维护闭环 |

IEC 61131-3 runtime、在线修改和认证协议栈的成熟度必须独立评估，不能由
`process_sequence` 组件等级代替。

### 7.2 移动平台与多传感融合

| 阶段 | 技术范围 |
|------|----------|
| `E1` | 受控路线/姿态上的标定、时间对齐、基本融合和闭环跟踪 |
| `V2` | GNSS 丢失、轮滑、振动、磁干扰、时间戳跳变、传感器重连、坡地和载荷变化 |
| `I3` | 多车辆/多安装/宽温/寿命数据、自动标定、故障隔离、现场地图与标签闭环 |

导航前端达到 `I3` 不代表 SLAM、路径规划或整车安全达到相同等级。

### 7.3 协议适配与确定性 QoS

| 阶段 | 技术范围 |
|------|----------|
| `E1` | 记录报文回放下的分类、优先级、有界队列、超时和基础转发 |
| `V2` | 峰值突发、乱序、丢包、时钟漂移、背压、总线离线/恢复和最坏延迟测量 |
| `I3` | 多厂商互操作、长时间 soak、异常流量、配置升级回滚、现场拥塞和丢包闭环 |

只有 QoS/适配组件可按此晋级；EtherCAT、PROFINET、CAN 安全层等协议符合性
仍需对应协议组织、测试套件和产品认证。

### 7.4 生物信号与医疗前端

| 阶段 | 技术范围 |
|------|----------|
| `E1` | 公开或研究数据上的滤波、质量指数和特征有效，明确非诊断用途 |
| `V2` | 目标 AFE、导联/光路、运动伪迹、丢样、宽温、功耗、实时性和设备数据验证 |
| `I3` | 仅工程组件可在声明采集包络内标记；临床算法还需临床数据、风险管理和法规流程 |

框架测试不能将 ECG/PPG 算法自动转化为医疗诊断能力，植入式设备也不进入
普通 `I3` 晋级路径。

### 7.5 高安全与高吞吐场景

ASIL-D 制动/转向、SIL4 轨交、植入式医疗、完整 5G 基带、高分辨率视频和
高速金属增材协调，不设置“依靠本框架晋级 I3”的路线。此类项目只能：

- 复用经过隔离的非安全数学组件；
- 将 Zeplod 放在非安全监测、记录或辅助 MCU；
- 由系统级安全分析、冗余硬件、认证工具链和独立验证决定可用范围。

## 8. 实施优先级

建议按“复用价值 × 与当前硬件能力距离 × 可验证性”推进：

1. `control_loop`、`sensor_quality`、`fault_derating`；
2. 电机、数字电源、BMS、运动控制达到 `E1 → V2`；
3. `bm_stream`、`spectral_diagnostics`、DAQ/振动样例；
4. 音频、医疗信号前处理、超声前端；
5. `estimation_fusion`，落地飞控/AGV/农机前端；
6. `process_sequence`，落地 HVAC/灌溉/半导体辅助控制；
7. `transport_qos`，用于协议栈适配，但不自研完整工业协议；
8. 低分辨率视觉与 TinyML 前后处理；
9. 高安全等级和高吞吐场景仅做边界清晰的子系统验证。

领域数量不能替代领域深度。优先把少数 `A` 类组件推进到真实 `V2`，再扩展
更多 `B/C` 类场景，比同时提供大量停留在 `D0/E1` 的算法名称更有价值。
