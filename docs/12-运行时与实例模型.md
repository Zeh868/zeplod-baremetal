# 12 运行时与实例模型

> **本文职责**：单 MCU 单运行时设计、查询 API 与多 `bm_exec` 模型。

本文说明框架**单 MCU 单运行时**设计，回应对「单实例全局状态」的架构质疑。该模型是刻意取舍，不是实现疏漏。

## 设计原则

| 原则 | 说明 |
|------|------|
| 每 MCU 一套运行时 | 裸机固件通常单地址空间、单入口；无需进程级多实例 |
| 静态分配 | 所有队列、表、槽位在编译期或 `init` 时绑定，无堆 |
| 多控制回路 | 用多个 `bm_exec_t` + `bm_resource` 声明，而非多套 `bm_event` |
| 可观测 | 各子系统提供只读状态查询，供测试与监督逻辑审计 |

## 子系统范围

| 子系统 | 静态状态 | 多实例替代方案 |
|--------|----------|----------------|
| `bm_event` | 全局事件总线 | 按事件类型分区；跨域用 `bm_snapshot` |
| `bm_hrt` | 单 Scheduled 调度器 | 多槽 + `bm_exec` 绑定 |
| `bm_ticker` | 单毫秒发布器 | 多 slot |
| `bm_exec` | 单批次会话 + 实例表 | 表内多 `bm_exec_t`（≤ `BM_CONFIG_MAX_EXEC_INSTANCES`） |
| `bm_sync` | 单活动同步域 | 切换域须 `safe_stop` / 重配 |
| `bm_module` / `bm_wdg` | 单表 | 表内多模块条目 |

头文件索引：`include/bm_runtime_model.h` 中 `bm_runtime_subsystem_t`。

## 会话与查询 API

| API | 用途 |
|-----|------|
| `bm_exec_get_session()` | 控制批次 `NONE / INITED / STARTED / STOPPING` |
| `bm_hrt_is_started()` | Scheduled HRT 是否已启动 |
| `bm_ticker_is_initialized()` | Ticker 是否已 init |
| `bm_sync_get_state()` | 同步域 `IDLE / TRANSITION / CONFIGURED / ARMED` |
| `bm_event_reset()` 等 | 测试与启动时确定性复位 |

测试用例应在 `setUp()` 中调用相应 `reset` / `safe_stop`，保证用例隔离。

## 与 SIL-2 的关系

功能安全语境下，**固定内存映射的单运行时**便于：

- FMEA 中界定状态所有者
- 禁止隐式动态分配
- WCET 分析时不引入第二套并发调度器

若产品需要物理隔离的多运行时（例如双核 asymmetric MP），应在核间边界上拆分镜像，而非在同一核上实例化两套 `bm_event`。
