## 1. 术语对照

文档与代码里「HRT」一词有两层含义，容易混淆：

| 叫法 | 代码标识 | 触发源 | 执行上下文 |
|------|----------|--------|------------|
| **Hardware HRT**（常简称 HSRT） | `BM_CTRL_SLOT_HARDWARE` | PWM 更新、ADC 完成、比较器 IRQ | HAL 在中断里**直接**调回调 |
| **Scheduled HRT** | `BM_CTRL_SLOT_SCHEDULED` → `bm_hrt` | 专用定时器 tick（如 TIM6） | `bm_hrt` ISR 扫描到期槽 |
| **SRT** | 主循环 | 无硬件截止 | `bm_event_process` / `bm_ticker_poll` |

**关键实现事实**（读源码时务必记住）：

- `bm_hrt` **只分发** `BM_HRT_TRIGGER_TIMER` 的槽；`bm_hrt.c` 中 `hrt_dispatch()` 会跳过其他触发类型。
- **Hardware HRT 不经过 `bm_hrt`**。`bm_ctrl_init_all()` 通过 `bind_hardware()` 把 HAL IRQ 绑到 `bm_ctrl_run_binding()`，再由它调用你的 `step` 函数。
- `bm_hrt_trigger_t` 枚举里的 `PWM_UPDATE` / `ADC_COMPLETE` 用于 **slot 描述**；Hardware 路径由 HAL 绑定实现，Scheduled 路径只用 `TIMER`。

```text
                    ┌──────────────────────────────────────┐
  ADC/PWM IRQ ─────►│ Hardware HRT                         │
                    │  HAL → bm_ctrl_run_binding → step    │
                    └──────────────────────────────────────┘

  TIM6 tick   ─────►│ Scheduled HRT (bm_hrt)               │
                    │  hrt_dispatch → callback → step      │
                    └──────────────────────────────────────┘

  main loop   ─────►│ SRT                                  │
                    │  bm_ticker_poll / bm_event_process   │
                    └──────────────────────────────────────┘
```

## 2. 为什么要分域

裸机机电控制里同时存在两类矛盾需求：

1. **硬截止任务**：电流环必须在 PWM 周期内完成，WCET 以微秒计。
2. **软任务**：通信协议、Shell、显示、复杂状态机，允许毫秒级抖动。

若共用同一条执行路径和同一套数据结构（事件队列、内存池、日志），会出现：

- HRT ISR 里 `malloc` / 遍历订阅表 → WCET 不可分析；
- SRT 持锁时被高优先级 HRT 抢占 → 数据结构不一致；
- 用事件队列传递 20 kHz 采样流 → 队列爆满、延迟不可控。

框架的解法是 **静态分区**：按时间约束划分执行域，并规定跨域只能通过专用通道通讯。

### 2.1 域边界规则

| 规则 | 说明 |
|------|------|
| HRT ISR 禁止 | `bm_event_process`、订阅回调、`bm_mempool_alloc`（常规路径）、`bm_channel`、`BM_LOG` |
| HRT ISR 允许 | 读 ADC、写 PWM/比较器、写 `bm_snapshot`、调 `bm_event_publish_copy_from_isr`（仅入队，不执行回调） |
| SRT 允许 | 事件、池、通道、Shell、看门狗、读 `bm_snapshot` |
| 亚微秒保护 | 比较器、定时器 Break 等硬件路径，不依赖 CPU 调度 |

`bm_critical` 保护的是 **SRT 共享结构**。HRT 路径不应长时间持锁；分析 WCET 时须计入 Hardware HRT 对 Scheduled HRT 的抢占。

## 3. Hardware HRT 用法

### 3.1 直接 HAL 绑定（不用 `bm_ctrl_inst`）

适合单环、资源极简的场景。关键要弄清：**`bind_complete` 把函数指针存进 HAL；向量 ISR 在硬件到来时取出该指针并调用**。二者通过 HAL 内部的 `g_adc_complete_binding`（名称因平台略异）连在一起。

#### 绑定物：`bm_hal_hrt_binding_t`

HAL 与应用只约定一个「接线盒」——两个字段，无魔法：

```c
typedef struct bm_hal_hrt_binding {
    void (*callback)(void *context);   /* 环算法入口 */
    void *context;                     /* 传给 callback 的用户状态 */
} bm_hal_hrt_binding_t;
```

`bm_hal_adc_bind_complete(adc, &binding)` 的作用就是：**把这份 `binding` 拷进 HAL 静态变量，并打开 ADC 完成中断**。之后每次硬件 JEOC，向量 ISR 都从**同一份**静态变量里取 `callback` 来调。

#### 完整关联（应用代码 ↔ HAL 代码）

下面左右对照的是 **同一仓库** `platform/backends/register_stm32g4/bm_drv_adc_stm32g4.c` 中的真实逻辑（仿真版 `native_sim` 同理，只是用 `fire_complete()` 代替硬件 IRQ）。

**① 应用：定义环算法 + 注册绑定（上电 / init 时执行一次）**

```c
/* 你的电流环——将来会被 ISR 间接调用 */
static void current_loop_step(void *ctx) {
    motor_state_t *m = (motor_state_t *)ctx;
    uint16_t adc;
    bm_hal_adc_read_injected(&BM_HAL_ADC1, 0, &adc);
    m->duty = pi_update(&m->pi, adc);
    bm_hal_pwm_set_duty(&BM_HAL_PWM_TIM1, 0, m->duty);
}

void motor_hrt_bind(void) {
    bm_hal_hrt_binding_t binding = {
        .callback = current_loop_step,   /* ①a 函数地址交给 HAL */
        .context  = &motor_state,        /* ①b 状态指针交给 HAL */
    };
    bm_hal_adc_bind_complete(&BM_HAL_ADC1, &binding);  /* ①c 触发 HAL 保存 */
}
```

**② HAL：`bind_complete` 把 ① 存入静态变量并开中断**

```c
static bm_hal_hrt_binding_t g_adc_complete_binding;  /* HAL 内部的「接线盒」 */

int bm_hal_adc_bind_complete(const bm_hal_adc_t *adc,
                             const bm_hal_hrt_binding_t *binding) {
    /* ... 校验 adc ... */
    g_adc_complete_binding = *binding;   /* ② 拷贝 ①a/①b：callback + context */
    BM_ADC_IER(...) |= BM_ADC_IER_JEOCIE; /* 使能注入完成中断 */
    return BM_OK;
}
```

此时内存里等价于：`g_adc_complete_binding.callback == current_loop_step`，`g_adc_complete_binding.context == &motor_state`。

**③ 硬件：注入 ADC 随 PWM/定时器触发转换，完成后置位 JEOC**

（板级 init 配置触发源，不由 `bind_complete` 完成。）

**④ HAL：向量 ISR 取出 ② 中保存的指针并调用 ① 的函数**

```c
void ADC1_2_IRQHandler(void) {           /* 启动文件向量表指向此符号 */
    if ((BM_ADC_ISR(...) & JEOC) == 0) return;
    BM_ADC_ISR(...) &= ~JEOC;            /* 清硬件标志 */
    if (g_adc_complete_binding.callback) /* ④ 读 ② 里存的函数指针 */
        g_adc_complete_binding.callback(
            g_adc_complete_binding.context);  /* → 进入 current_loop_step(&motor_state) */
}
```

**⑤ 应用：`current_loop_step` 在 IRQ 上下文跑完电流环后返回 → ISR 返回**

#### 时序（把上面 ①–⑤ 串起来）

```mermaid
sequenceDiagram
    participant App as 应用 init
    participant HAL as HAL bind_complete
    participant Box as g_adc_complete_binding
    participant HW as ADC 硬件
    participant ISR as ADC1_2_IRQHandler
    participant Loop as current_loop_step

    App->>Loop: 定义环算法
    App->>HAL: bind_complete(adc, {callback, context})
    HAL->>Box: 保存 callback / context
    HAL->>HW: 使能 JEOC 中断
    HW-->>ISR: 转换完成，进中断
    ISR->>Box: 读取 callback
    ISR->>Loop: callback(context)
    Loop->>Loop: 读 ADC、PI、写 PWM
    Loop-->>ISR: 返回
    ISR-->>HW: 中断返回
```

#### 仿真环境（无硬件 IRQ）

`native_sim` 没有 `ADC1_2_IRQHandler`，但 **同一份** `complete_binding` 仍在 `bind_complete` 时写入。测试或示例里调用：

```c
bm_hal_adc_sim_fire_complete(&BM_HAL_ADC_SIM0);
```

内部等价于 ④：`complete_binding.callback(complete_binding.context)`。因此 `hrt_servo_stub` 主循环里手动 `fire_complete` 也能跑通电流槽。

#### 解绑与注意

- `bm_hal_adc_bind_complete(adc, NULL)`：关 JEOC 中断并清空 `g_adc_complete_binding`。
- `bind_complete` **不**初始化 ADC 时钟、注入序列、PWM 同步触发；这些在板级 `init` 完成。
- 环函数在 IRQ 上下文执行，禁止 `BM_LOG`、`bm_event_process` 等 SRT 操作。

更多 HAL 契约见 [08-HAL移植指南](08-HAL移植指南.md)。

### 3.2 通过 `bm_ctrl_inst`（推荐，多实例）

与 §3.1 **中断对接方式相同**：最终仍调用 `bm_hal_adc_bind_complete`，仍由 HAL 向量 ISR 调 `g_adc_complete_binding.callback`。差别是框架把 `callback` 固定为 `bm_ctrl_run_binding`，再由它转调你的 `slot->step`：

```text
ADC1_2_IRQHandler
    → g_adc_complete_binding.callback   /* 框架填入 bm_ctrl_run_binding */
        → binding->slot->step(instance) /* 你的 current_step */
```

在实例的 `slots[]` 中声明 `BM_CTRL_SLOT_HARDWARE`：

```c
static const bm_ctrl_slot_t g_servo_slots[] = {
    {
        BM_CTRL_SLOT_HARDWARE,
        0u,                              /* period_us 对硬件槽无效 */
        BM_HRT_TRIGGER_ADC_COMPLETE,
        current_step,                    /* 你的环算法 */
        bind_adc,                        /* 绑 HAL */
        "current"
    },
};
```

`bind_adc` 典型实现：

```c
static int bind_adc(const bm_ctrl_inst_t *inst,
                    const bm_hal_hrt_binding_t *binding) {
    (void)inst;
    return bm_hal_adc_bind_complete(&BM_HAL_ADC0, binding);
}
```

`bm_ctrl_init_all()` 在 `bind_hardware` 时传入的 `hal_binding` 形如：

```c
hal_binding.callback = bm_ctrl_run_binding;
hal_binding.context  = &g_bindings[i];   /* 含 instance + slot 指针 */
slot->bind_hardware(inst, &hal_binding);
```

`bm_ctrl_run_binding` 内部调用 `binding->slot->step(binding->instance)`。组装全流程见 [§8](#8-bm_ctrl_inst-组装流程)。

## 4. Scheduled HRT（`bm_hrt`）

与 §3.1 同一模式：**`bm_hrt_init` 把 callback 写入 `g_slots[]`；定时器 ISR 扫描到期槽并调用**。不经过 Hardware 的 `bind_complete`。

### 4.1 接线（应用 ↔ `bm_hrt.c`）

**① 应用注册槽并启动**

```c
static void speed_loop(void *ctx) {
    axis_t *a = (axis_t *)ctx;
    /* 速度环 */
}

static const bm_hrt_slot_t slots[] = {
    { .period_us = 1000, .trigger = BM_HRT_TRIGGER_TIMER,
      .callback = speed_loop, .context = &axis, .name = "speed" },
};
bm_hrt_init(slots, 1);    /* ① 拷贝进 g_slots[]，计算 next_tick */
bm_hrt_start();           /* ② 启动 HAL 定时器，注册 hrt_timer_isr */
```

**② 框架保存槽表**（`bm_hrt_init`）

```c
g_slots[i].pub = slots[i];
g_slots[i].period_ticks = slots[i].period_us / BM_CONFIG_HRT_TICK_US;
g_slots[i].next_tick = now + period_ticks;
```

**③ 定时器 ISR → 分发**（`bm_hrt_start` 注册 `hrt_timer_isr`）

```c
static void hrt_timer_isr(void) {
    hrt_dispatch();
}

static void hrt_dispatch(void) {
    for (each slot in g_slots) {
        if (now >= slot->next_tick)
            slot->pub.callback(slot->pub.context);  /* → speed_loop(&axis) */
            slot->next_tick += period_ticks;
    }
}
```

```text
bm_hrt_init({ callback=speed_loop })
    → g_slots[]
bm_hrt_start()
    → bm_hal_timer_set_callback(hrt_timer_isr)
TIM IRQ → hrt_timer_isr → hrt_dispatch → speed_loop(context)
```

### 4.2 约束

- `period_us` 须整除 `BM_CONFIG_HRT_TICK_US`，换算后不超过 `INT32_MAX` tick。
- 槽数 ≤ `BM_CONFIG_HRT_MAX_SLOTS`。
- tick 比较支持 `uint32_t` 自然回绕；单次调度间隔必须小于半个计数空间。
- 错过 deadline 时饱和递增 `bm_hrt_get_deadline_missed()` / `bm_hrt_get_deadline_missed_total()` 计数，并触发 `bm_hrt_deadline_missed_hook`；钩子在 ISR 中执行，默认空实现，覆盖实现不得记录日志；每轮 ISR 每槽最多补跑 1 次 callback。

### 4.3 经 `bm_ctrl_inst`

`BM_CTRL_SLOT_SCHEDULED` 槽在 `init_all` 时并入 `g_hrt_slots[]`，`callback` 固定为 `bm_ctrl_run_binding`，`context` 指向 `g_bindings[]`。`bm_ctrl_start_all()` 在**全部**实例 `ops->start` 成功后才开放会话门并调用 `bm_hrt_start()`；Scheduled 与 Hardware `step` 都仅在会话 `STARTED` 后执行。独立使用 `bm_hrt` 时仍须手动启动。见 [§8](#8-bm_ctrl_inst-组装流程)。

## 6. 同步域 `bm_sync`

### 6.1 解决什么问题

同步域解决 **多路 PWM/ADC 的硬件时序对齐**，例如：

- 多关节机器人：各轴同时起步；
- 交错并联电源：各相固定相位差；
- 多 ADC 注入序列在同一 master 相位触发。

**同步域不负责数据通讯**。电流、位置、电压等仍走 `bm_snapshot` 或事件。

### 6.2 描述符

```c
typedef struct {
    const char *name;
    const bm_hal_timer_t *master_timer;   /* 主定时器（TRGO 等） */
    const bm_ctrl_inst_t *const *members; /* 参与同步的实例 */
    const uint32_t *phase_ticks;          /* 各成员相对主相位的 tick 偏移 */
    uint32_t member_count;
} bm_sync_domain_t;
```

### 6.3 状态机与 `trigger` 实际作用

```text
configure → HAL 配主从定时器/相位，不输出
arm       → 武装，仍不启动
trigger   → HAL 按 phase_ticks 同时释放各成员 PWM/定时器（one-shot，须重新 arm）
safe_stop → 停机并清状态
```

框架层校验：`member_count` ≤ `BM_CONFIG_MAX_SYNC_MEMBERS`；各 `phase_ticks[i]` ≤ `BM_CONFIG_SYNC_MAX_PHASE_TICKS`（主定时器 tick 偏移上界，与 HAL 主周期契约配合使用）。

**`trigger` 不直接调用你的 `step` 函数**。以 `multi_axis_sync` 为例：

```text
各轴 axis_step 挂在 BM_CTRL_SLOT_SCHEDULED（1ms）
bm_hrt_start 后，Scheduled ISR 周期调用 axis_step

sync_trigger 的作用：让各轴 PWM/计数器在同一硬件相位同时「开跑」
→ 各轴第一次进入 axis_step 时 bm_hal_timer_get_ticks() 相同
```

真机 HAL：配置 master TRGO、从定时器相位寄存器等。`native_sim` 的 `bm_sync_hal_native` 仅置标志，无真实 PWM，但示例仍用 **同 tick 进入 `axis_step`** 验证逻辑。

框架层（`bm_sync.c`）校验描述符、拒绝重复成员、维护 `g_active_domain`/`g_armed`，委托 `bm_sync_hal_*`。任一 configure/arm/trigger HAL 失败都会调用 `bm_sync_hal_safe_stop` 并清除活动状态。

### 6.4 调用顺序

```c
bm_ctrl_init_all(instances, count);
bm_sync_configure(&domain);
bm_sync_arm(&domain);
bm_ctrl_start_all(instances, count); /* 内部在全部 start 后再 bm_hrt_start */
bm_sync_trigger(&domain);
```

未链平台 sync HAL（`bm_sync_hal_stub.c`）时 `configure` 失败；PC 须链 `bm_sync_hal_native.c`。

### 6.5 与数据通讯的边界

同步域只解决**何时一起起步/相位**；电流、位置等仍用 `bm_snapshot` 或实例 `state`（见 [§7](#7-跨域通讯)）。

## 7. 跨域通讯

### 7.1 选型矩阵

| 方向 | 传什么 | 推荐机制 | 不推荐 |
|------|--------|----------|--------|
| HRT → SRT | 最新测量值、状态字 | `BM_SNAPSHOT_PUBLISH` / `READ` | `bm_event` 高频捎带 |
| HRT(快) → HRT(慢) | 降采样结果 | 同上 `bm_snapshot` | 共享无保护全局变量 |
| ISR → SRT | 「发生了」+ 小 payload | `bm_event_publish_copy_from_isr` | 在 ISR 里调订阅者 |
| SRT → SRT | 业务消息 | `bm_event` / `bm_channel` | — |
| SRT → HRT | 设定值、使能位 | 反向 snapshot 或单写者 `volatile` 字段 | SRT 持锁写复杂结构 |

**原则**：事件是 **信号**（通知刷新），不是 **数据管道**。实时流走 snapshot。

### 7.2 `bm_snapshot` 三缓冲原理

头文件组件 `include/bm_snapshot.h`，无 `.c` 实现。

```c
BM_SNAPSHOT_DEFINE(pack_current, float);
static pack_current_snapshot_t g_pack = BM_SNAPSHOT_INITIALIZER;
```

**写者**（HRT，如 ADC 完成）：

```c
float amps = ...;
BM_SNAPSHOT_PUBLISH(g_pack, &amps);
```

**读者**（SRT 或慢 HRT）：

```c
float amps;
BM_SNAPSHOT_READ(g_pack, &amps);
```

内部逻辑：

1. 三份 `buffer[3]`，维护 `published`（最新完成帧）与 `reading`（读者占用）。
2. 写者选「既不是 published 也不是 reading」的槽写入，内存屏障后 flip `published`。
3. 读者标记 `reading`，拷贝，再释放。若写者在此期间再次发布，读者可能重试（`do-while` 保证一致性拷贝）。

特性：

- 无锁，适合单写者单读者；
- 允许丢中间帧，读者始终拿到 **最新完成的一帧**；
- 适合传感器、电流、位置等「最新值即可」的量。

### 7.3 示例对照

见 [07 §6](07-应用骨架与数据流.md#6-示例数据流速查)。

### 7.4 SRT → HRT 反向路径

文档与示例较少涉及，常见模式：

1. **反向 snapshot**：SRT `PUBLISH` 设定值，HRT `READ`（注意 WCET：HRT 读路径须极简）。
2. **单写者约定**：SRT 写 `volatile` 标量或对齐的结构，HRT 只读；启用前用 `bm_memory_barrier_release()`。
3. **使能/模式**：通过 `bm_ctrl_inst` 的 `ops->start` / `safe_stop` 在 SRT 侧切换，而非在环内改复杂状态机。

设计时须明确 **谁写、谁读、是否允许丢帧**。

## 8. `bm_ctrl_inst` 组装流程

`bm_ctrl_init_all()` 把各实例槽位接到 §3.1 / §4.1 的两条 HRT 路径：

```text
assemble_tables()
  对每个 slot：
    g_bindings[i] = { instance, slot }
    若 SCHEDULED：
      g_hrt_slots[j] = { period_us, TIMER, bm_ctrl_run_binding, &g_bindings[i] }
bm_hrt_init(g_hrt_slots, ...)          → §4 的 g_slots[]
各实例 ops->init()
对每个 HARDWARE slot：
  hal_binding = { bm_ctrl_run_binding, &g_bindings[i] }
  slot->bind_hardware(inst, &hal_binding)  → §3 的 g_adc_complete_binding 等
```

| API | 作用 |
|-----|------|
| `bm_ctrl_init_all` | 校验、组表、`bm_hrt_init`、硬件 bind；会话 → INITED |
| `bm_ctrl_start_all` | 各实例 `ops->start`（使能外设），全部成功后再 `bm_hrt_start`；会话 → STARTED |
| `bm_hrt_start` | 启动 Scheduled ISR；通常由 `start_all` 自动调用 |
| `bm_ctrl_safe_stop_all` | 关闭会话门、停 HRT、解绑 Hardware、逆序 safe_stop、`bm_hrt_reset`；会话 → NONE |

伺服三环与仿真 `fire_complete` 见 [07 §6](07-应用骨架与数据流.md#6-示例数据流速查)。CMake/`bm_config.h` 见 [05](05-构建配置与CMake.md)。
