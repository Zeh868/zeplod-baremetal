# 10 迁移与演进

> **本文职责**：Ultra → Core → 混合域 → Zephyr 的升级路径与 API 差异。

Zeplod Baremetal 设计为可逐级升级：芯片变强时加组件，产品进机器人栈时迁 Zephyr。

## 演进路线图

```text
[Ultra]  ──升级 MCU──►  [Core]  ──要模块化──►  [Core + Module]
                                                    │
                    ──要硬实时控制──►  [+ HRT / Ctrl / Sync]
                                                    │
                    ──要 OS / 网络──►  [Zephyr + Zeplod]
```

## 迁移时注意

1. **先跑通 `native_sim` 示例**，再迁 HAL。
2. **保持 `bm_config.h` 有界**：迁 Zephyr 前避免无上限配置习惯。
3. **事件 ID** 与上层 Zeplod 栈对齐时，集中定义在 `app_events.h`。

## Ultra → Core

### 事件系统对照

| Ultra | Core |
|-------|------|
| `BM_ULTRA_EVENT_BIND(EVENT, cb)` | `bm_event_register_type(EVENT, "name"); bm_event_subscribe(EVENT, cb, ud, &id);` |
| `bm_ultra_publish(type, data, len)` | `bm_event_publish_copy(type, prio, data, len)` |
| 无优先级 | 显式 `bm_event_priority_t`（0=最高） |
| 无 user_data | 订阅时传入 `user_data` |
| FIFO | 优先级扫描出队 |
| 单回调 | 多订阅者链表 |

### 代码示例

```c
/* Ultra */
static const bm_ultra_callback_t _bm_ultra_callbacks[8] = {
    BM_ULTRA_CB(EVENT_TEMP, on_temp)
};

void on_temp(const void *data, uint8_t len) {
    uint16_t val = *(const uint16_t *)data;
}
bm_ultra_publish(EVENT_TEMP, &val, sizeof(val));
```

```c
/* Core */
bm_event_register_type(EVENT_TEMP, "TEMP");
bm_event_subscriber_id_t id;
bm_event_subscribe(EVENT_TEMP, on_temp, NULL, &id);

void on_temp(const bm_event_t *event, void *user_data) {
    uint16_t val = *(const uint16_t *)event->data;
}
bm_event_publish_copy(EVENT_TEMP, BM_EVENT_PRIO_NORMAL, &val, sizeof(val));
```

### 数据所有权

- **Ultra**：值拷贝到队列内联数组，无需管理内存。
- **Core**：`publish_copy` 由框架拷贝（小 payload 可内联）；`publish_event` 由调用方保证生命周期。

## Core → Core + Module

将分散的 `init`/主循环逻辑收进 `bm_module` 表，统一 `init_all` / `start_all` 与失败回滚。

示例：`core_sensor`、`full_system`。机制说明见 [04-核心机制](04-核心机制.md)。

## Core → 混合域（Control）

### 1. 启用 CMake 选项

```cmake
set(BM_ENABLE_HRT ON)
set(BM_ENABLE_TICKER ON)
set(BM_ENABLE_EXEC ON)   # 可选
set(BM_ENABLE_SYNC ON)        # 可选，依赖 EXEC
```

### 2. 配置 `bm_config.h`

增加 `BM_CONFIG_HRT_TICK_US`、`BM_CONFIG_HRT_MAX_SLOTS`、`BM_CONFIG_MAX_EXEC_INSTANCES` 等（见 `bm_config.h.template`）。

### 3. 初始化顺序

```c
bm_event_reset();
bm_hal_timer_init(tick_hz);            /* 与 HRT/ticker 共用自由运行计数器 */
bm_exec_init_all(instances, count);    /* 可选：组 HRT 表、bind Hardware */
bm_sync_configure/arm(&domain);        /* 可选：可在 start_all 前或后 */
bm_exec_start_all(instances, count);   /* 可选：全部 ops->start 后再 bm_hrt_start */
bm_sync_trigger(&domain);              /* 可选：one-shot，重复触发须 re-arm */
bm_ticker_init(ticker_slots, count);   /* 可选：须在 subscribe 与定时器就绪后 */

for (;;) {
    bm_ticker_poll();
    bm_event_process();
}
```

不经 `bm_exec` 时改为 `bm_hrt_init` + `bm_hrt_start`。详见 [07 §5.2](07-应用骨架与数据流.md#52-正确初始化顺序)。

### 4. 架构边界

- 划分 Hardware HRT / Scheduled HRT / SRT 任务（见 [02-混合域与执行模型](02-混合域与执行模型.md)）。
- HRT ISR **不调用** `bm_event_*`。
- 跨域数据用 `bm_snapshot`，不用事件队列传高频采样。
- 用 `bm_resource` 声明外设，避免多实例冲突。
- 默认 `BM_CONFIG_ENABLE_PRIORITY_MASK=0` 可保持原有临界区行为。

## Baremetal → Zephyr

### 模块生命周期

| Baremetal | Zephyr |
|-----------|--------|
| `BM_MODULE_DEFINE` + `BM_MODULE_TABLE` | `module_register()` |
| `bm_module_init_all()` | `module_init_all()` |
| `bm_module_start_all()` | `module_start_all()` |
| `bm_module_stop_all()` | `module_stop_all()` |

### 事件系统

回调签名一致：`void cb(const event_t *event, void *user_data)`

| Baremetal | Zephyr |
|-----------|--------|
| `bm_event_publish_copy()` | `event_publish_copy()` |
| `bm_event_subscribe()` | `event_subscribe()` |
| `bm_event_register_type()` | `event_register_type()` |

### 数据通道（不兼容）

`bm_channel` 是 SPSC 数据通道，**不是** Zeplod Data Bus。升级到 Zephyr 后需改用 `data_bus` API（命名 channel、多消费者、引用计数、retain/release）。

硬实时部分可能下沉到 Zephyr 驱动，或保留在协处理器裸机固件中。
