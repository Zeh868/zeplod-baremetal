# 08 迁移与演进

Zeplod Baremetal 设计为可逐级升级：芯片变强时加组件，产品进机器人栈时迁 Zephyr。

## 演进路线图

```text
[Ultra]  ──升级 MCU──►  [Core]  ──要模块化──►  [Core + Module]
                                                    │
                    ──要硬实时控制──►  [+ HRT / Ctrl / Sync]
                                                    │
                    ──要 OS / 网络──►  [Zephyr + Zeplod]
```

## Ultra → Core

| 变化 | 说明 |
|------|------|
| 事件绑定 | 编译期表 → `register_type` + `subscribe` |
| 发布 | `bm_ultra_publish` → `bm_event_publish_copy` + 优先级 |
| 订阅者 | 单回调 → 多订阅者 |
| 内存 | 无池 → `bm_mempool` |

逐步对照与代码示例：[migration/ultra-to-core.md](migration/ultra-to-core.md)。

## Core → Core + Module

将分散的 `init`/主循环逻辑收进 `bm_module` 表，统一 `init_all` / `start_all` 与失败回滚。示例：`core_sensor`、`full_system`。

## Core → 混合域（Control）

1. CMake 打开 `BM_ENABLE_HRT`、`BM_ENABLE_TICKER`、`BM_ENABLE_CTRL_INST`。
2. 划分 Hardware HRT / Scheduled HRT / SRT 任务（见 [02-混合域与执行模型](02-混合域与执行模型.md)）。
3. 用 `bm_resource` 声明外设，避免多实例冲突。
4. 跨域数据改用 `bm_snapshot`，不用事件队列传高频采样。

API 与启动顺序变化：[migration/core-to-hybrid.md](migration/core-to-hybrid.md)。

## Baremetal → Zephyr

- 模块与业务逻辑大部分可复用。
- 事件语义对齐 Zeplod 事件总线；`bm_channel` 需换为 Zeplod Data Bus。
- 硬实时部分可能下沉到 Zephyr 驱动或保留协处理器固件。

详情：[migration/core-to-zephyr.md](migration/core-to-zephyr.md)。

## 迁移时注意

1. **先跑通 `native_sim` 示例**，再迁 HAL。
2. **保持 `bm_config.h` 有界**：迁 Zephyr 前避免无上限配置习惯。
3. **事件 ID** 与上层 Zeplod 栈对齐时，集中定义在 `app_events.h`。
